# ADR-001: JWT over Session Cookies for Authentication
# Intelligent Academic Advisor — MVP
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-010, REQ-011, REQ-003, REQ-041

---

## Context

The system requires counselors to authenticate before accessing student data (REQ-010). The three-tier architecture (REQ-003) means the React frontend and FastAPI backend are separate processes — the frontend is served as a static bundle from a browser, and the backend is a distinct HTTP service. An authentication mechanism must work cleanly across this boundary.

Two options were evaluated:

**Option A — Server-side session cookies:**
The backend stores session state (e.g., in memory or a Redis store) and issues an opaque session ID via a `Set-Cookie` header. The browser transmits the cookie automatically on subsequent requests. The backend looks up the session ID on every request.

**Option B — JWT (JSON Web Tokens):**
The backend signs a self-contained token containing the user's identity (`user_id`) and an expiry claim. The token is returned in the login response body. The frontend stores it in memory and attaches it explicitly to every request as an `Authorization: Bearer` header. The backend validates the token signature on every request without any server-side lookup.

---

## Decision

**JWT (Option B) is adopted.**

---

## Reasoning

1. **Statelessness fits the three-tier model.** The FastAPI backend is designed as a stateless API server (REQ-003). JWT validation requires no shared state store — the `SECRET_KEY` environment variable is sufficient. Session cookies would require either an in-process session store (which does not survive restarts) or an external store such as Redis, adding infrastructure complexity not warranted for MVP.

2. **No real-time collaboration removes the need for session management.** The system explicitly excludes real-time collaboration (REQ-041). There is no requirement to invalidate another user's session on a state change. The absence of server-side session state is therefore not a liability.

3. **Explicit header attachment avoids CSRF.** JWTs carried in the `Authorization` header are not sent automatically by the browser. This eliminates the class of Cross-Site Request Forgery (CSRF) vulnerabilities that require additional mitigation when using cookies.

4. **Simplicity.** JWT issuance and validation are well-supported by FastAPI's ecosystem (python-jose, PyJWT). No additional infrastructure (Redis, session database table) is required.

5. **No role complexity.** REQ-011 prohibits role-based access control in MVP. The JWT payload carries only `sub` (user UUID), `exp`, and `iat`. There is no need for complex claim structures that might favour a session-based approach.

6. **Extensibility.** If the system later requires role-based claims or multi-tenant scoping (REQ-007), additional claims can be added to the JWT payload without changing the authentication infrastructure.

---

## Consequences

**Accepted trade-offs:**

- **No server-side logout.** Because tokens are stateless and no blacklist is maintained, a token remains cryptographically valid until its `exp` claim elapses, even if the counselor clicks "logout." The frontend mitigates this by discarding the token from memory. The short expiry window (30 minutes in production) limits the risk.
- **Re-login on page refresh.** Storing the token in React context (in-memory) means it is lost when the page is refreshed. This is acceptable for an internal tool used by counselors in active sessions (REQ-001).
- **Token cannot be revoked immediately.** If a token is compromised, the only remediation is to rotate the `SECRET_KEY`, which invalidates all active tokens and forces all counselors to re-login. A token blacklist would resolve this but is out of scope for MVP.

**Not affected:**
- The API contract is unchanged regardless of token mechanism; the `Authorization: Bearer` header is standard.
- Frontend pages are protected by React routing logic that checks for token presence in context; this does not change with JWT vs cookies.
