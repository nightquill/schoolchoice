# ADR-004: API Versioning Strategy (/api/v1/)
# Intelligent Academic Advisor — MVP
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-003, REQ-007, REQ-042, REQ-023

---

## Context

The system exposes a REST API consumed exclusively by the React frontend. An API versioning strategy must be chosen before the first endpoint is built, because changing the base path retroactively would require coordinated changes across every API call in the frontend codebase.

Options evaluated:

**Option A — No versioning:**
Paths take the form `/students/{id}`, `/auth/login`, etc. No version segment is included.

**Option B — URL path versioning:**
A version segment is embedded in the path: `/api/v1/students/{id}`, `/api/v1/auth/login`. The version can be incremented to `/api/v2/...` when breaking changes are introduced.

**Option C — Header versioning:**
The API version is supplied by the caller in a custom HTTP header (e.g., `Accept: application/vnd.advisor.v1+json` or `X-API-Version: 1`). The path does not contain a version segment.

**Option D — Query parameter versioning:**
The version is passed as a query parameter (e.g., `/students?version=1`).

---

## Decision

**URL path versioning with the prefix `/api/v1/` is adopted (Option B).**

---

## Reasoning

1. **Explicitness and discoverability.** The version is visible in the URL. Any developer, tool, or log entry can immediately identify which version of the API is being called without inspecting headers or query parameters. This aligns with REQ-009's principle of transparency and interpretability applied to system interfaces as well as outputs.

2. **Unambiguous routing.** FastAPI's router groups can be mounted under `/api/v1/`, making version-based routing a structural property of the application rather than a runtime decision. This requires no conditional logic in individual endpoint handlers.

3. **Extensibility.** REQ-007 requires the architecture to support future enhancements without rewriting MVP components. When breaking changes are required (e.g., restructuring the recommendation response schema to support ML-based scoring), a new `/api/v2/` router can be added and run in parallel. The frontend and any future consumers can migrate incrementally while `/api/v1/` remains stable.

4. **Industry standard.** URL path versioning is the most widely adopted convention for REST APIs and the approach most developers expect. It requires no special documentation to explain.

5. **No external consumers in MVP.** REQ-023 prohibits external API integrations. The sole consumer of the API is the internal React frontend. This removes the most common argument for header-based versioning (backward-compatible negotiation with heterogeneous clients). However, the `/api/v1/` prefix still provides value as a forward-looking design choice.

6. **The `/api/` prefix separates API routes from potential static asset routes.** Although the frontend is served as a static bundle separately from the FastAPI server, the `/api/` prefix makes it unambiguous that a path is handled by the backend API, not a file server or reverse proxy serving static assets. This simplifies reverse proxy configuration in production.

7. **Build order alignment.** REQ-042 prescribes a build order in which the backend API is built before the frontend. Establishing the versioned path convention at backend build time ensures the frontend is written against the correct, stable URL structure from the start.

---

## Consequences

**Accepted trade-offs:**

- **Slightly longer paths.** All API paths are prefixed with `/api/v1/`. This is a minor aesthetic inconvenience with no functional impact.
- **Version increment requires a decision process.** When a breaking change is needed, a new version must be explicitly created rather than silently mutating the existing contract. This is intentional and desirable: it forces deliberate communication of breaking changes.
- **Header and query-parameter versioning are excluded.** Option C (header versioning) would make API calls harder to test in a browser address bar and harder to reason about from logs. Option D (query parameter versioning) pollutes the query string and can interfere with caching. Neither provides meaningful benefit over URL path versioning for this system.

**Version increment criteria (guidance for future iterations):**
A new version (`/api/v2/`) is warranted when a change to an existing endpoint's request or response schema would break a deployed frontend without modification. Adding new endpoints or new optional fields to existing responses does not require a version increment.
