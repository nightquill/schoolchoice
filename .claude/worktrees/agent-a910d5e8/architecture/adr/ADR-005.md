# ADR-005: Async Plan Generation — Background Task with Polling
# Intelligent Academic Advisor — v2 Pipeline
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-049, REQ-078, REQ-099, REQ-105

---

## Context

The v2 AcademicPlan generates a rich, 7-section HTML document (REQ-077). This
document incorporates: the student's full grade profile, eligibility analysis against
every target school, a gap analysis table per school, SHAP-based rationale paragraphs
(when ML model is active), a quarterly action timeline, and language readiness
assessment. Generating this content involves multiple database reads and non-trivial
string composition.

The preferences.md SLA (REQ-049) specifies: all read API responses must complete in
under 500 ms. Plan generation is explicitly exempted and may take up to 10 seconds.

The system must therefore not block the HTTP response thread during plan generation.

Three approaches were evaluated:

---

## Options Evaluated

### Option A — Synchronous generation (blocking)
The POST /plan endpoint generates the plan inline, holds the HTTP connection open,
and returns the completed plan in a single response.

**Rejected because:**
- Generation can take several seconds. Holding the connection is fragile: proxy
  timeouts (typically 30–60 s in nginx/cloud load balancers) can silently drop the
  request mid-generation. The frontend receives a 502 or connection reset with no
  recoverable state.
- The frontend's request timeout must be set unreasonably high (> 10 s) for any
  endpoint other than plan generation, which would hide real performance problems.
- A single slow generation blocks the ASGI worker for its duration. Under FastAPI's
  default worker model this reduces throughput for all concurrent users.

### Option B — FastAPI BackgroundTasks (in-process asyncio)
The POST /plan endpoint accepts the request, inserts a job record, dispatches an
asyncio background task via FastAPI's `BackgroundTasks`, and returns 202 Accepted
immediately. The frontend polls a separate status endpoint until the job is complete.

**Selected — see Decision below.**

### Option C — Celery + Redis/RabbitMQ worker queue
The POST /plan endpoint enqueues a Celery task. A separate Celery worker process
consumes the queue and performs generation. Status is tracked in Celery's result
backend (Redis).

**Rejected for MVP because:**
- Introduces two new infrastructure components (Celery worker process, Redis broker)
  that must be added to Docker Compose, increasing deployment complexity for a
  single-server MVP (REQ-003 and REQ-049 emphasise simplicity).
- Celery's task serialisation and result backend add round-trip latency and failure
  modes that are disproportionate to the scale (one user, one plan at a time).
- FastAPI's built-in BackgroundTasks covers the requirement with zero additional
  infrastructure.
- Celery is documented as the correct upgrade path if multi-process scaling is needed
  in the future (see Consequences below).

---

## Decision

**FastAPI BackgroundTasks (Option B) is adopted for MVP plan generation.**

The transcript parsing async flow (REQ-067, REQ-106) uses the same pattern for
consistency.

**Pattern specification:**

1. POST /students/{id}/plan:
   - Validate the request (JWT, ownership, profile completeness).
   - Insert a `plan_generation_jobs` row with `status = "pending"`.
   - Register the generation coroutine with FastAPI `BackgroundTasks`.
   - Return 202 Accepted with `{job_id, status: "pending", submitted_at}`.

2. Background task (runs after response is sent):
   - Update job to `status = "running"`.
   - Read all required student data from the database.
   - Run the Plan Generator module (produces HTML string).
   - On success: UPSERT `academic_plans` (incrementing `version`), update job to
     `status = "complete"`, set `completed_at`.
   - On exception or timeout: update job to `status = "failed"`,
     set `error_message`, set `completed_at`.

3. GET /students/{id}/plan/status:
   - Read `plan_generation_jobs` row and return its current status fields.
   - The frontend polls this endpoint; recommended polling interval: 2 seconds.
   - When `status == "complete"`, the frontend issues GET /students/{id}/plan.

4. Timeout enforcement:
   - The background task wraps the generation call in `asyncio.wait_for()` with
     `PLAN_GENERATION_TIMEOUT_SECONDS` as the deadline.
   - `asyncio.TimeoutError` is caught and mapped to a `"failed"` job status.

---

## Reasoning

1. **No new infrastructure.** FastAPI's BackgroundTasks runs in the same ASGI process.
   Docker Compose does not need a new service. This is decisive for a single-school
   MVP deployment.

2. **Immediate response.** The 202 pattern returns immediately, preserving the
   sub-500 ms SLA for the trigger endpoint itself.

3. **Idempotent polling.** The status endpoint is a simple read; it can be called
   any number of times without side effects. The frontend can implement exponential
   backoff trivially.

4. **Failure visibility.** Job status is persisted in the database. If the backend
   restarts mid-generation, the job record remains in `"running"` state. A startup
   reconciliation step can detect stale running jobs and mark them `"failed"`,
   prompting the user to retry.

5. **Upgrade path preserved.** The Plan Generator module is called through a clean
   function boundary. Replacing BackgroundTasks with a Celery enqueue call in the
   future requires changing only the dispatch line, not the generator itself.

---

## Consequences

**Accepted trade-offs:**

- **In-process tasks can be interrupted by process restart.** If the FastAPI process
  is killed while a task is running, the job record stays `"running"`. A startup
  reconciliation job (scheduled or on-boot) must detect and reset these to `"failed"`.
  This is acceptable at MVP scale.

- **No parallelism across workers.** If the backend is scaled to multiple ASGI
  workers (e.g., multiple Gunicorn workers), a background task runs only in the
  worker that accepted the POST request. At MVP single-server scale this is not
  a problem. Multi-worker scale requires Celery (see upgrade path).

- **Upgrade path to Celery:** When the system grows to require multiple backend
  processes or dedicated worker nodes, replace the BackgroundTasks dispatch with a
  Celery task enqueue. The `plan_generation_jobs` table schema remains unchanged;
  Celery writes the same `status` transitions.

- **Transcript parse uses the same pattern.** POST /transcript returns 202 with a
  `job_id`. Parse status is tracked on the `transcripts.parse_status` field (no
  separate jobs table needed since there is one transcript record per upload event).
