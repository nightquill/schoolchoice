# ADR-006: SHAP for ML Explainability
# Intelligent Academic Advisor — v2 Pipeline
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-046, REQ-075, REQ-102

---

## Context

The v2 matchmaking engine includes an optional XGBoost classifier that produces an
admission probability score for each student–school pair (REQ-074). The system must
explain to counsellors and students why a particular school scored as it did
(REQ-075). This explanation must be presented as the top 3 contributing features in
plain English on the Student Target Schools page and within the Academic Plan
(REQ-102).

Explainability is a first-class requirement, not an afterthought. The system is
used by counsellors and parents who need to understand and trust the recommendations.
An opaque score would undermine adoption (see REQ-009 from v1, which established
the principle of transparent scoring).

Three approaches to feature importance / explainability were evaluated:

---

## Options Evaluated

### Option A — Model feature importance (global)
XGBoost natively exposes `feature_importances_` — the average gain or split count
per feature across all trees, computed once at training time. This gives a global
ranking of which features matter most to the model overall.

**Rejected because:**
- Global importance tells you which features are important on average across all
  students and schools. It cannot explain why a specific student–school pair scored
  as it did. Telling a student "aggregate score is generally the most important
  factor" is not the same as "your aggregate score of 22 is 3 points below this
  school's average admitted score, which reduced your probability by 18 percentage
  points."
- Per-prediction context is lost.

### Option B — LIME (Local Interpretable Model-agnostic Explanations)
LIME fits a local linear surrogate model around each prediction by perturbing the
input and observing the output.

**Rejected because:**
- LIME's approximations can be unstable: small changes to input can produce
  noticeably different explanation orderings. This instability would confuse users
  who compare plans across sessions.
- LIME requires significantly more computation per prediction than SHAP (many
  perturbation samples).
- SHAP has superseded LIME as the industry standard for tree-based models and has
  better theoretical grounding (Shapley values from cooperative game theory).

### Option C — SHAP (SHapley Additive exPlanations)
SHAP decomposes each prediction into per-feature contributions, where each
contribution is the marginal effect of including that feature (averaged over all
possible feature coalitions). For tree ensembles, SHAP's `TreeExplainer` computes
exact Shapley values in polynomial time (not approximate).

**Selected — see Decision below.**

---

## Decision

**SHAP with `shap.TreeExplainer` is adopted for per-prediction explainability.**

**Implementation specification:**

1. At match time, after XGBoost produces `predict_proba()` for a student–school pair:
   - Instantiate `shap.TreeExplainer(model)` (cached at startup, not per prediction).
   - Call `explainer.shap_values(feature_vector)` to obtain the contribution of
     each feature to this specific prediction.
   - Identify the top 3 features by absolute SHAP value.
   - Map each feature to a plain-English sentence using a lookup template:
     - `aggregate_score`: "Your aggregate score ({value}) is {above/below} the
       school's average admitted score ({threshold})."
     - `ielts_score`: "Your IELTS score ({value}) {meets/falls short of} the
       requirement ({threshold})."
     - `extracurricular_count`: "Your {value} extracurricular activities {strengthen/
       weaken} your profile relative to this school's typical applicants."
     - (and so on for each feature in the feature vector)
   - Store the top 3 as JSONB on `StudentSchoolTarget.shap_explanation`:
     ```json
     {
       "top_features": [
         {"feature": "aggregate_score", "value": 22, "direction": "negative",
          "plain_text": "Your aggregate score (22) is below the school's average admitted score (26)."},
         ...
       ]
     }
     ```

2. `shap_explanation` is stored at match time and served directly by the GET
   /targets and GET /match endpoints. No SHAP computation occurs at read time.

3. When the ML model is not loaded (`ML_MODEL_PATH` unset), SHAP is not computed
   and `shap_explanation` remains null. The weighted scoring component still provides
   a one-sentence rationale derived from the dominant scoring dimension.

---

## Reasoning

1. **Theoretical soundness.** Shapley values satisfy four axioms (efficiency, symmetry,
   dummy, additivity) that ensure the contributions are mathematically fair and sum
   to the predicted output. This gives a principled basis for communicating why a
   specific score was reached.

2. **Exact for tree models.** `shap.TreeExplainer` computes exact Shapley values for
   XGBoost in O(TLD) time (T = trees, L = leaves, D = depth) without approximation.
   This is faster and more accurate than LIME or sampling-based methods at this
   feature dimensionality (< 20 features).

3. **Per-prediction, not global.** Each student–school pair gets a distinct
   explanation grounded in the specific feature values for that pair. This is the
   meaningful unit of explanation for counsellors.

4. **Industry standard.** SHAP is the de facto standard for tree model explainability
   in production systems. The `shap` Python library is maintained, well-documented,
   and integrates directly with XGBoost.

5. **Plain-English mapping is backend-controlled.** Feature names and their plain-
   English templates are maintained in a single backend configuration file. Updating
   the language of explanations does not require model retraining.

---

## Consequences

**Accepted trade-offs:**

- **SHAP computation adds latency per match run.** At the expected feature
  dimensionality (< 20 features) and model size (hundreds of trees), TreeExplainer
  shap_values() is typically < 10 ms per prediction. For a student with 20 target
  schools, total SHAP time is < 200 ms, well within the overall match budget.

- **SHAP is only meaningful when the ML model is active.** When falling back to
  weighted scoring, the system must still provide some rationale; this is derived
  from the dominant scoring dimension (e.g., "Academic fit score of 0.72 is the
  strongest matching factor").

- **Plain-English templates require maintenance.** As features are added or renamed,
  the template mapping must be updated. This is a documentation and maintenance
  discipline item, not a technical risk.

- **Storing top 3 only.** Full SHAP value arrays (one value per feature) are not
  stored in the database to avoid bloat. If full arrays are needed for model
  debugging, they can be recomputed from the stored feature vector. The feature
  vector is not currently stored; if this audit need arises, the schema should be
  extended to store it.
