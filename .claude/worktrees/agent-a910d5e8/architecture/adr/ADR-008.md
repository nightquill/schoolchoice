# ADR-008: Data Encryption Approach — Full-Disk vs Column-Level
# Intelligent Academic Advisor — v2 Pipeline
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-050

---

## Context

The system stores student PII (Personally Identifiable Information) in the
PostgreSQL database. Fields requiring protection include:
- `date_of_birth`
- `address`
- `phone`
- `email` (student contact email; distinct from account login email)

The preferences.md (§13) states: "All user data must be encrypted at rest." The
architect must evaluate PostgreSQL column-level encryption versus full-disk encryption
and document the decision.

This decision affects both the operational security posture and the implementation
complexity for the backend engineer. It must be made before the database schema is
built.

---

## Options Evaluated

### Option A — Full-Disk Encryption (FDE)

Full-disk encryption encrypts the entire storage volume (database files, WAL logs,
temporary files, backups) at the OS or filesystem layer. The database engine reads
and writes plaintext; encryption and decryption are transparent and handled by the
storage layer (e.g., Linux LUKS, macOS FileVault, cloud provider volume encryption
such as AWS EBS encryption at rest, GCP Persistent Disk encryption).

**How it works:**
- The host OS mounts an encrypted volume. All writes to the volume are encrypted
  before hitting physical storage. All reads are decrypted transparently.
- The PostgreSQL process and all application code interact with plaintext data.
  No changes to SQL, ORM models, or query logic are required.

### Option B — Column-Level Encryption (CLE)

Column-level encryption encrypts specific field values before they are written to
the database and decrypts them on read. This can be implemented at the application
layer (Python code encrypts before INSERT, decrypts after SELECT, using a key from
a secret store) or via PostgreSQL's `pgcrypto` extension
(`pgp_sym_encrypt` / `pgp_sym_decrypt`).

**How it works:**
- The backend holds an encryption key (a separate `ENCRYPTION_KEY` environment
  variable, never stored in the database).
- On INSERT/UPDATE: sensitive fields are encrypted before being passed to the ORM.
- On SELECT: sensitive fields are decrypted after retrieval.
- The database stores ciphertext. Even a full database dump reveals no plaintext
  PII.

---

## Decision

**Full-disk encryption (Option A) is adopted for MVP. Column-level encryption
(Option B) is documented as the recommended future hardening path.**

---

## Reasoning

### Why Full-Disk Encryption for MVP

1. **Zero application code changes.** FDE is transparent to the backend, the ORM,
   and all SQL queries. No schema changes, no new Python dependencies, no test
   cases for encryption/decryption logic. This preserves the backend engineer's
   velocity.

2. **Covers the primary threat model.** The main attack vector for a single-school
   web application is physical disk theft or unauthorised access to backup files
   (e.g., a database dump that escapes the server). FDE mitigates both. If an
   attacker obtains the disk or backup file without the encryption key, the data
   is unreadable.

3. **No key management burden at the application layer.** FDE keys are managed by
   the OS or cloud provider. In a Docker Compose deployment on a cloud VM (the
   expected MVP environment), enabling volume encryption is a single checkbox in
   the cloud console. The application does not need to manage, rotate, or expose
   any encryption keys.

4. **Backup coverage.** FDE protects database backups automatically because backups
   are written to encrypted volumes or, in cloud provider managed backup services,
   encrypted independently. Column-level encryption also protects backups (the
   ciphertext is backed up), but this is a secondary consideration — FDE is simpler
   to verify.

5. **Docker Compose constraint (REQ-003, REQ-049).** The system is designed for a
   single-server Docker Compose deployment. Introducing application-layer CLE would
   require the backend to manage a separate encryption key, add `cryptography` or
   `pgcrypto` dependencies, and test encryption round-trips. This complexity is not
   justified at MVP scale.

### Why Column-Level Encryption is the Future Hardening Path

CLE provides a stronger defence-in-depth posture that FDE alone cannot:

1. **Database-level access control.** FDE protects against offline disk attacks but
   not against an authorised database user querying PII directly. A compromised
   application database credential exposes all plaintext data under FDE. Under CLE,
   a database dump or direct SQL query returns ciphertext — usable only by code
   that holds the encryption key.

2. **Insider threat and credential compromise.** If a developer with database access
   inadvertently exposes a credential (e.g., in a log, in a crash dump), CLE limits
   the blast radius to non-PII fields.

3. **Regulatory alignment.** If the system moves to a regulated environment
   (Hong Kong PDPO, GDPR for international expansion), column-level encryption of
   named PII fields provides auditable, field-specific protection that regulators
   can inspect.

**Recommended CLE implementation when hardening:**
- Add `ENCRYPTION_KEY` to backend environment variables (32-byte random hex).
- Use the `cryptography` Python library (Fernet symmetric encryption) to
  encrypt/decrypt `date_of_birth`, `address`, `phone`, `email` in a dedicated
  PII mixin applied to the Student ORM model.
- Store encrypted fields as `TEXT` (base64-encoded ciphertext) in the database.
- Treat the encryption key as a first-class secret, rotated via a key rotation
  procedure (re-encrypt all affected rows, then invalidate the old key).
- No changes to the API contracts are required; encryption is transparent to callers.

---

## Threat Model Summary

| Threat | FDE Coverage | CLE Coverage |
|--------|-------------|--------------|
| Physical disk theft | Protected | Protected |
| Database backup exfiltration | Protected (if backup volume is encrypted) | Protected |
| Compromised DB credentials (external attacker) | Not protected | Protected |
| Malicious insider with DB access | Not protected | Protected |
| Compromised application process (code execution) | Not protected | Not protected* |
| Encrypted volume key theft (OS-level) | Not protected | Not applicable |

*Neither FDE nor CLE protects against a fully compromised application process that
holds the decryption key in memory. This threat requires additional controls (key
management services, hardware security modules) beyond the scope of MVP.

---

## Consequences

**Accepted trade-offs:**

- **FDE does not protect against DB credential compromise.** This is an accepted
  risk at MVP scale. Mitigation: strong database passwords, no publicly reachable
  database port (enforced by Docker Compose network topology), JWT-protected API.

- **FDE is infrastructure-dependent.** The security posture assumes the cloud
  provider's volume encryption is correctly configured. The deployment checklist
  (to be maintained by the operations role) must include verification of volume
  encryption status.

- **Migrating from FDE to CLE later requires a data migration.** When CLE is
  adopted, all existing plaintext PII must be encrypted in a migration script.
  The schema migration (TEXT field type change) is non-breaking. The migration
  script must be run in a transaction with rollback capability.

- **No changes required to the current database schema or ORM for MVP.** The
  `date_of_birth`, `address`, `phone`, and `email` fields are stored as plaintext
  strings. The CLE migration changes their storage type and adds encryption/
  decryption logic; the API contracts are unaffected.
