## 1. Entity Definitions

### 1.1 User

Represents an authenticated counselor account.

**REQ-IDs:** REQ-010, REQ-011, REQ-024, REQ-028

| Field | Type | Nullable | Notes |
|-------|------|----------|-------|
| id | UUID | No | Primary key |
| email | string | No | Unique; used as login credential |
| hashed_password | string | No | Bcrypt hash; plaintext is never stored |
| created_at | timestamp with timezone | No | Set at INSERT; not modifiable |
| updated_at | timestamp with timezone | No | Set at INSERT and UPDATE |

**Constraints:**
- `email` must be unique across all users.
- `hashed_password` is never returned in any API response.

---

### 1.2 Student

Represents a student profile managed by a counselor.

**REQ-IDs:** REQ-012, REQ-013, REQ-014, REQ-015, REQ-025, REQ-028, REQ-033

| Field | Type | Nullable | Notes |
|-------|------|----------|-------|
| id | UUID | No | Primary key |
| user_id | UUID | No | Foreign key → users.id; cascade delete |
| name | string | No | Student full name |
| grades | JSONB | No | Map of subject (string) to grade value (string). Example: `{"math": "A", "english": "B+"}` |
| interests | JSONB | No | Array of interest tag strings. Example: `["robotics", "music"]` |
| strengths_weaknesses | text | No | Free-text; may be an empty string but not NULL |
| target_region | string | No | Constrained to `"local"` or `"international"` |
| created_at | timestamp with timezone | No | Set at INSERT |
| updated_at | timestamp with timezone | No | Set at INSERT and UPDATE |

**Constraints:**
- `user_id` must reference an existing user; row is deleted when the owning user is deleted.
- `target_region` is an enum with exactly two valid values.

---

### 1.3 School

Represents a school stored in the system's internal database.

**REQ-IDs:** REQ-026, REQ-030

| Field | Type | Nullable | Notes |
|-------|------|----------|-------|
| id | UUID | No | Primary key |
| name | string | No | Unique school name |
| location | string | No | Geographic location string |
| min_academic_requirements | JSONB | No | Map of subject (string) to minimum grade (string). Example: `{"math": "B", "english": "C+"}` |
| key_strengths | JSONB | No | Array of strength tag strings. Example: `["STEM", "arts"]` |
| notes | text | Yes | Free-text notes; nullable |
| created_at | timestamp with timezone | No | Set at INSERT |
| updated_at | timestamp with timezone | No | Set at INSERT and UPDATE |

**Constraints:**
- `name` must be unique across all schools.

---

### 1.4 Recommendation

Represents a single matched school result for a student, generated by the matching engine.

**REQ-IDs:** REQ-016, REQ-017, REQ-018, REQ-019, REQ-020, REQ-027, REQ-029, REQ-037

| Field | Type | Nullable | Notes |
|-------|------|----------|-------|
| id | UUID | No | Primary key |
| student_id | UUID | No | Foreign key → students.id; cascade delete |
| school_id | UUID | No | Foreign key → schools.id; no cascade (school deletion is independent) |
| school_name | string | No | Denormalized copy of school name at time of generation |
| score | float | No | Computed match score in range 0.0–1.0 |
| explanation | text | No | Plain-text rationale: which factors contributed and their weight |
| gaps | text | No | Plain-text description of student deficiencies relative to school requirements |
| rank | integer | No | Ordinal rank within this student's recommendation set (1 = highest); constrained 1–5 |
| created_at | timestamp with timezone | No | Set at INSERT; each generate run replaces all prior records |

**Constraints:**
- At most 5 recommendation rows per student at any time (REQ-019). The POST recommendations endpoint deletes existing rows before inserting new ones.
- `rank` is 1-based and unique per student.

---

### 1.5 ActionPlan

Represents the action plan generated for a student.

**REQ-IDs:** REQ-021, REQ-022, REQ-038

| Field | Type | Nullable | Notes |
|-------|------|----------|-------|
| id | UUID | No | Primary key |
| student_id | UUID | No | Foreign key → students.id; cascade delete; unique (one plan per student) |
| academic_targets | text | No | Plain-text academic improvement targets |
| extracurricular_direction | text | No | Plain-text suggested extracurricular focus |
| preparation_steps | text | No | Plain-text general preparation guidance |
| created_at | timestamp with timezone | No | Set at INSERT |
| updated_at | timestamp with timezone | No | Set at UPDATE (replace on re-generate) |

**Constraints:**
- `student_id` is unique (one ActionPlan per Student). Re-generating an action plan replaces the existing row rather than inserting a new one (UPSERT).

## 2. Entity Relationship Summary

```
users (1) ──────────── (many) students
students (1) ─────────── (many) recommendations
students (1) ─────────── (0 or 1) action_plans
recommendations (many) ─────── (1) schools
```

- REQ-028: One user owns many students.
- REQ-029: One student has zero to many recommendation records (zero before first generate run, up to 5 after).
- One student has zero or one action plan.
- Each recommendation references one school; school deletion does not cascade-delete recommendations (the denormalized `school_name` field preserves display integrity).

## 3. Data Flow by Operation

### 3.1 Counselor Registration

```
FRONTEND                BACKEND                  DATABASE
  │                        │                         │
  │ POST /auth/register     │                         │
  │ {email, password} ────►│                         │
  │                        │ Hash password            │
  │                        │ INSERT users ──────────►│
  │                        │◄────── {id, email, ...} │
  │◄── 201 {id, email} ────│                         │
```

Sensitive path: `hashed_password` is stored in the database and never returned to the frontend.

---

### 3.2 Counselor Login

```
FRONTEND                BACKEND                  DATABASE
  │                        │                         │
  │ POST /auth/login        │                         │
  │ {email, password} ────►│                         │
  │                        │ SELECT users            │
  │                        │ WHERE email=? ─────────►│
  │                        │◄──────── users row       │
  │                        │ Verify password hash     │
  │                        │ Sign JWT                 │
  │◄── 200 {access_token} ─│                         │
  │                        │                         │
  │ Store token in memory  │                         │
```

The JWT carries the `user_id` in its claims. The frontend stores the token in memory (not localStorage; see auth_spec.md).

---

### 3.3 Create Student Profile

```
FRONTEND                BACKEND                  DATABASE
  │                        │                         │
  │ POST /students          │                         │
  │ Authorization: Bearer   │                         │
  │ {name, grades, ...} ──►│                         │
  │                        │ Validate JWT → user_id  │
  │                        │ INSERT students         │
  │                        │ (user_id from JWT) ────►│
  │                        │◄──────── students row    │
  │◄── 201 {student} ──────│                         │
```

The `user_id` is taken from the validated JWT, never from the request body. This enforces the ownership constraint (REQ-028).

### 3.5 Trigger Matching and Generate Recommendations

This is the central workflow (REQ-040).

```
FRONTEND                BACKEND                  DATABASE
  │                        │                         │
  │ POST /students/{id}/   │                         │
  │   recommendations      │                         │
  │ Authorization: Bearer ─►│                         │
  │                        │ Validate JWT → user_id  │
  │                        │ SELECT students         │
  │                        │ WHERE id=? ────────────►│
  │                        │◄──────── student row     │
  │                        │ Verify ownership        │
  │                        │                         │
  │                        │ SELECT schools ────────►│
  │                        │◄──────── [all schools]   │
  │                        │                         │
  │                        │ ┌─────────────────────┐ │
  │                        │ │ Matching Engine      │ │
  │                        │ │ 1. Filter by grades  │ │
  │                        │ │ 2. Score (3 factors) │ │
  │                        │ │ 3. Rank top 5        │ │
  │                        │ └─────────────────────┘ │
  │                        │                         │
  │                        │ DELETE recommendations  │
  │                        │ WHERE student_id=? ────►│
  │                        │                         │
  │                        │ INSERT recommendations  │
  │                        │ (up to 5 rows) ────────►│
  │                        │◄──────── [rec rows]      │
  │                        │                         │
  │◄── 201 [{rec}, …] ─────│                         │
```

The matching engine operates entirely in backend memory. It reads from the database but does not write intermediate state. It writes only the final top-5 results.

## 4. Data that Crosses Component Boundaries

### What the Frontend sends to the Backend:
- `Authorization: Bearer <JWT>` header on every protected request.
- Student profile fields: `name`, `grades`, `interests`, `strengths_weaknesses`, `target_region`.
- School record fields: `name`, `location`, `min_academic_requirements`, `key_strengths`, `notes`.
- Login credentials: `email`, `password` (over HTTPS only).
- Empty POST bodies to trigger matching and action plan generation.

### What the Backend returns to the Frontend:
- `access_token`, `token_type`, `expires_in` on login.
- Student objects (all fields except `user_id` is included for ownership awareness; `hashed_password` is never included).
- School objects (all fields).
- Recommendation objects: `id`, `student_id`, `school_id`, `school_name`, `score`, `explanation`, `gaps`, `rank`, `created_at`.
- Action plan objects: `id`, `student_id`, `academic_targets`, `extracurricular_direction`, `preparation_steps`, `created_at`, `updated_at`.

### What the Backend stores in the Database:
- All entity rows as defined in §1 above.
- `hashed_password`, never plaintext.
- `grades`, `interests`, `min_academic_requirements`, `key_strengths` as JSONB columns.
- `explanation` and `gaps` as computed plain-text strings generated by the matching engine.
- `academic_targets`, `extracurricular_direction`, `preparation_steps` as computed plain-text strings.

### What never crosses component boundaries:
- Plaintext passwords are never stored in the database and never appear in API responses.
- Raw SQL is never exposed outside the backend.
- The `SECRET_KEY` used to sign JWTs is a backend environment variable only; it is never sent to the frontend.
- No data flows to external systems (REQ-023).
