**Verification approach:** After all changes, run the full test suite. The tests import from `app.db.models` and `app.db.models_v2` — the re-export stubs make both paths resolve to the new module location. No test file modifications needed.

**Critical ordering:** The import chain is:
1. `app.db.models` defines Base, User
2. `app.modules.school_choice.models.models` imports Base from app.db.models, defines domain models
3. `app.db.models` re-exports Student, School, etc. from the module
4. `app.db.models_v2` re-exports v2 models from the module

This creates a circular import risk between `app.db.models` and `app.modules.school_choice.models.models`. To avoid this: the re-exports in `app.db.models` must be at the BOTTOM of the file, after User class definition. Python resolves this because by the time the re-export line executes, Base and User are already defined.

If circular import occurs, the fallback is: move the re-exports to a separate file `app.db._compat.py` and have both `models.py` and `models_v2.py` use lazy imports. But try the direct approach first.
