## 1. System Classification

**System Type:** Content Generation (YAML-driven structured generation)

**Description:**
Template-driven AI consultant engine that transforms entity data + ML recommendations into professional plans/presentations. Each deployment defines domain-specific "consultant tasks" via YAML — the engine assembles data, runs a single structured AI call returning validated JSON, and renders a professional HTML report with charts and actionable roadmaps via Jinja2. Used by non-technical users (parents, counselors) who click "Generate Plan" and receive a professional deliverable. "Good" means: structurally valid JSON matching the task schema, all sections populated with entity-specific content, factually grounded in the provided data, and rendered as a polished HTML report.

**Critical Failure Modes:**
1. AI returns malformed JSON that fails schema validation — user sees error instead of plan
2. AI fabricates data not present in the entity record (e.g., invents school programs that don't exist)
3. Streaming SSE connection drops mid-generation — user sees partial/broken output
4. AI ignores entity-specific data and produces generic boilerplate recommendations
5. Confidence badges misrepresent data completeness — user trusts a recommendation based on insufficient data

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** K-12 Education / School Choice Advisory

**User Population:** Parents and guardians navigating K-12 school selection for their children; secondary users are school counselors and education consultants who review or co-present these plans with families. Users are non-technical and treat the generated plan as a professional advisory document — comparable to a real estate agent's comparative market analysis or a financial planner's portfolio recommendation.

**Stakes Level:** High

**Output Consequence:** Families use the generated school choice plan to decide which schools to apply to, which programs to pursue, and how to prioritize their application effort. A flawed plan — one that misranks schools, omits eligibility disqualifiers, or presents fabricated program details — can cause a family to invest application effort in schools the student cannot attend, miss schools that are actually good fits, or make enrollment decisions based on incorrect information. Downstream harms include wasted application fees, missed enrollment windows (which often cannot be reopened), and misplaced confidence in a recommendation the data does not support.

### What Domain Experts Evaluate Against

```
Dimension: School eligibility accuracy
Good: Every school listed in the plan is one the student qualifies to apply to based on the
      district's actual admission rules (grade band, attendance zone, language program
      eligibility, lottery availability, etc.)
Bad: Plan recommends a school that requires a minimum GPA the student does not meet, or
     lists a lottery school whose application window has closed.
Stakes: Critical
Source: School counselor professional standard — recommending an ineligible school wastes
        a family's time and erodes trust in the advisory tool immediately.
```

```
Dimension: Data grounding — no invented program details
Good: Every named program, extracurricular, or school feature cited in the plan exists in
      the school record provided to the system. Descriptions match the school's actual
      offerings without embellishment.
Bad: Plan mentions an "IB program" or "dual-language track" at a school that has no such
     program in the data, or describes a "small class size" that is not recorded anywhere
     in the entity record.
Stakes: Critical
Source: Education consultant professional standard — unsourced claims about a school's
        programs constitute the primary trust-destroying failure mode for AI advisory tools
        in education (arxiv.org LLM failure modes taxonomy; EdTech practitioner reports).
```

```
Dimension: Confidence transparency — badge matches actual data completeness
Good: A school marked HIGH confidence has complete grade records, attendance data, language
      assessment scores, and all eligibility-relevant fields populated. A school marked LOW
      confidence has a visible tooltip identifying the specific missing fields.
Bad: A school marked HIGH confidence is missing standardized test scores that the
     eligibility rule depends on — family invests effort based on a ranking that is
     materially undermined by missing data.
Stakes: High
Source: AI-09 requirement (eligibility confidence indicator); matchmaker_v2.py
        compute_data_completeness() implementation; practitioner standard that
        recommendation opacity is a primary parent complaint with algorithmic tools.
```

```
Dimension: Student-specificity — plan reads as written for this child, not a template
Good: The summary section names the student, references their specific grade level, notes
      their documented strengths (e.g., "strong math performance" when GPA data supports
      it), and ranks schools in order that reflects their actual match scores.
Bad: Plan uses generic language that would be equally true for any student ("schools in
     your area offer excellent programs"). Action items are generic (e.g., "visit school
     websites") rather than specific to the ranked list.
Stakes: High
Source: Parent/guardian usability standard — education consultants report that generic
        output is the single fastest way for a parent to lose confidence in an AI advisory
        tool. Framework for Coherent AI Use in K-12 Education (Child Trends, 2024).
```

```
Dimension: Action item completeness and sequence logic
Good: The plan's action items are ordered chronologically, reference actual deadlines (if
      present in school data), and distinguish between actions the family controls (submit
      application) vs. actions that are contingent (wait for lottery result).
Bad: Action items are listed in random order, reference deadlines that are not in the
     school data (fabricated), or omit a required step (e.g., pre-registration for a
     language assessment before a lottery).
Stakes: Medium
Source: School counselor workflow standard — counselors use action item lists as the
        primary handoff artifact to families; sequence errors cause families to miss steps.
```

### Known Failure Modes in This Domain

1. **Eligibility blind spots under sparse data.** When a student's record is missing fields that an eligibility rule depends on (e.g., no language assessment score for a dual-language program), the engine may rank the school as eligible by default rather than flagging it as unknown. Families interpret silence as confirmation of eligibility.

2. **Boilerplate drift under long prompts.** When the system prompt is long and student data is sparse, the model drifts toward generic advisory language indistinguishable between students. This failure is invisible without a per-student specificity check — the plan looks complete but contains no student-specific content.

3. **Confidence score inflation from partial records.** The confidence badge reflects data completeness on fields that exist in the record — it cannot detect fields that were never collected. A student with all collected fields populated but with key eligibility fields never collected can receive a misleadingly HIGH badge.

4. **Fabricated program details from model training data.** Large language models have extensive knowledge of school reputations and programs from their training data. They will generate plausible-sounding program descriptions for schools they "know about" even when the provided entity record is sparse — and the descriptions may not match the school's current offerings.

### Regulatory / Compliance Context

**FERPA (Family Educational Rights and Privacy Act):** Student academic records (grades, assessment scores, attendance, evaluation results) used to drive the recommendation engine are FERPA-protected education records. The platform operator (typically a school district or education consultant) is the data controller. The AI system processes these records to generate plans — this is a school official function under FERPA's legitimate educational interest provision, but:
- Student PII must not be transmitted to third-party AI providers in ways that give those providers independent rights to use or retain the data. LiteLLM's BYOK pattern (AI-03) and zero data-retention API agreements with providers mitigate this risk.
- Generated plans containing student PII are themselves education records subject to FERPA access and correction rights.

**COPPA (Children's Online Privacy Protection Act, 2025 amendments):** If the platform is accessed directly by or on behalf of children under 13, the 2025 COPPA amendments (effective 2025) require opt-in consent (shifted from opt-out), formal written data security programs, and strict data minimization. The school choice deployment targets parents/guardians as the primary user — students do not directly interact with the system — which reduces direct COPPA exposure, but operators must confirm this deployment assumption.

**State student privacy laws:** As of 2025, 121+ state laws impose additional student data privacy requirements beyond FERPA. Operators deploying in states with active student privacy legislation (California SOPIPA, New York Education Law 2-d, etc.) must confirm that AI-generated plan storage and any data shared with LLM providers complies with those state requirements.

**No FCA/HIPAA/financial regulation:** This is an education advisory tool, not a financial or medical system. Those regulatory frameworks do not apply.

## 2. Framework Decision

**Selected Framework:** No external AI framework — extend existing `call_ai()` pattern with LiteLLM

**Version:** LiteLLM >=1.40 (already integrated, Phase 2)

**Rationale:**
Phase 5 is structured content generation via a single LLM call returning validated JSON, rendered through Jinja2. The project already has the right architecture in `ai_service.py` + `plan_generator.py`. LiteLLM (integrated in Phase 2) provides full model-agnostic provider abstraction. Adding any agent framework would introduce abstraction overhead on top of a pattern the codebase already owns — increasing complexity, adding maintenance surface, and violating the "build first, external last" principle. The work is: add `call_ai_stream()` for SSE, a `task_engine.py` that loads YAML task definitions, and a `recommendation_engine.py` that evaluates YAML rule definitions. All extend existing code; none require a framework.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain | Abstraction overhead on top of a single-call pattern the codebase already owns. Would be reconsidered if future phases require chaining multiple AI calls with intermediate state. |
| LangGraph | Overkill — no stateful multi-step agent workflows in this phase. Single structured call pattern. |
| CrewAI | Multi-agent orchestration not needed — this is single-call content generation. |

**Vendor Lock-In Accepted:** No — LiteLLM provides model-agnostic abstraction already

### Entry Point Pattern — Streaming Plan Generation

This is the complete call chain for one "Generate Plan" click. It extends `ai_service.py` and adds the SSE endpoint. Copy-paste runnable given the existing project infrastructure.

```python
# ── Step 1: Add to backend/app/core/ai_service.py ──────────────────────────
from __future__ import annotations

import logging
from collections.abc import AsyncGenerator
from typing import Any

import litellm
from fastapi import HTTPException

from app.core.config import settings

logger = logging.getLogger(__name__)


async def call_ai_stream(
    messages: list[dict[str, str]],
    **kwargs: Any,
) -> AsyncGenerator[str, None]:
    """
    Async streaming variant of call_ai(). Yields SSE-formatted text chunks.
    Returns an async generator — wrap in FastAPI StreamingResponse.

    Raises HTTPException(503) if AI_API_KEY is not configured.
    Raises HTTPException(502) on unexpected provider error.
    """
    if not settings.AI_API_KEY:
        raise HTTPException(
            status_code=503,
            detail="AI chat is not available: AI_API_KEY is not configured.",
        )

    model_string = _build_model_string()  # reuse existing function in ai_service.py
    logger.info("Streaming AI call: %s", model_string)

    try:
        response = await litellm.acompletion(
            model=model_string,
            messages=messages,
            api_key=settings.AI_API_KEY,
            api_base=settings.AI_BASE_URL or None,
            timeout=settings.AI_TIMEOUT,
            stream=True,
            **kwargs,
        )
        async for chunk in response:
            content = chunk.choices[0].delta.content or ""
            if content:
                # SSE wire format — client reads event.data
                yield f"data: {content}\n\n"
    except litellm.AuthenticationError as exc:
        logger.error("AI auth failed: %s", exc)
        raise HTTPException(status_code=503, detail="AI provider authentication failed.")
    except litellm.ServiceUnavailableError as exc:
        logger.error("AI provider unreachable: %s", exc)
        raise HTTPException(status_code=503, detail="AI provider is temporarily unavailable.")
    except Exception as exc:
        logger.error("AI streaming error: %s", exc)
        raise HTTPException(status_code=502, detail="Unexpected error communicating with AI provider.")


# ── Step 2: SSE endpoint — backend/app/api/v1/routes/consultant.py ──────────
from fastapi import APIRouter, Depends, Query
from fastapi.responses import StreamingResponse
from sqlalchemy.orm import Session

from app.core.ai_service import call_ai_stream
from app.db.session import get_db
from app.platform.task_engine import TaskEngine

router = APIRouter()


@router.post("/consultant/tasks/{task_id}/stream")
async def stream_consultant_task(
    task_id: str,
    entity_id: str = Query(...),
    db: Session = Depends(get_db),
) -> StreamingResponse:
    """Execute a consultant task and stream AI output via SSE."""
    engine = TaskEngine()
    task_def = engine.load_task(task_id)
    messages = engine.build_messages(task_def, entity_id, db)

    return StreamingResponse(
        call_ai_stream(messages, max_tokens=task_def.max_tokens, temperature=task_def.temperature),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",   # disable nginx buffering — required for SSE in production
        },
    )
```

### Key Abstractions

| Abstraction | What It Is | When You Use It |
|-------------|-----------|-----------------|
| `call_ai(messages, **kwargs)` | Synchronous LiteLLM wrapper in `core/ai_service.py`. Returns the full response string. Handles provider routing, auth, and error normalisation. | Background plan generation stored to DB; plan chat edits (`plan_chat_service.py`); any case where the full JSON response is needed before any processing begins. Never bypass for direct `litellm.completion()` calls. |
| `call_ai_stream(messages, **kwargs)` | Async generator yielding `"data: <token>\n\n"` SSE chunks. To be added to `ai_service.py`. | User-facing "Generate Plan" button — users see text appearing progressively rather than waiting 10–30s for the full response. |
| `TaskEngine` | New `platform/task_engine.py`. Loads a task YAML, resolves named data slots (student/schools/matchmaker results), renders system and user prompts via Jinja2, then calls `call_ai()` or `call_ai_stream()`. | Every consultant task execution. One engine handles all task types — new tasks are new YAML files, not new Python classes. |
| `RecommendationEngine` | New `platform/recommendation_engine.py`. Loads a `rules.yaml`, evaluates eligibility filters and weighted scoring generically. Replaces the school-choice-specific logic embedded in `matchmaker_v2.py`. | Any domain that needs ranked recommendations with `compute_data_completeness()` confidence badges. |
| `yaml_loader.load_entity_yaml()` | Existing YAML-to-dataclass parser in `platform/yaml_loader.py`. Uses `yaml.safe_load()` with null guard and raises `ValueError` on missing required fields. | Extend this pattern for `load_task_yaml()` and `load_rules_yaml()` — never write raw `yaml.safe_load()` calls in feature code; always go through a typed loader function. |

### Common Pitfalls

1. **Calling `litellm.completion()` or `litellm.acompletion()` directly in feature code.** All LiteLLM calls must go through `call_ai()` or `call_ai_stream()` in `ai_service.py`. Bypassing them skips the API key guard, model string builder, provider-specific error normalisation, and any future observability hooks (tracing, cost tracking). Every `import litellm` outside `ai_service.py` is an architecture violation.

2. **Using `asyncio.run()` inside a FastAPI route or SSE generator.** FastAPI operates inside a running event loop. Calling `asyncio.run()` from within a coroutine raises `RuntimeError: This event loop is already running`. The fix is to define the route as `async def` and `await` async functions directly — never wrap them in `asyncio.run()`.

3. **Returning `StreamingResponse` without `X-Accel-Buffering: no`.** In any deployment with nginx in front of uvicorn (standard production setup), nginx buffers SSE by default. The user sees nothing until the full response is buffered and sent at once — the streaming UX is completely defeated. Always include `"X-Accel-Buffering": "no"` in SSE response headers.

4. **Attempting Pydantic validation on a streaming response.** `call_ai_stream()` yields raw text tokens as they arrive — the output is not valid JSON until the very last chunk. If the task requires Pydantic schema validation before the result is written to the database, use `call_ai()` (synchronous, returns the complete response) and validate after. Use `call_ai_stream()` only for the live display UX where the user watches generation. See Section 4b.2 for the complete decision table.

5. **Injecting unguarded entity data into prompts.** The Jinja2 prompt templates receive entity dicts that can grow unboundedly as the school database and student profile fields expand. A student with many subject grades, a full set of match results with SHAP explanation dicts, and all rationale text included can produce a user prompt exceeding 10K tokens. Always call `litellm.token_counter()` after building messages and apply the truncation strategy defined in Section 4 before calling the AI.

**Core Pattern:**

The implementing engineer must follow this call chain for every consultant task. No deviations from this structure.

```python
# backend/app/platform/task_engine.py
from __future__ import annotations

import json
import logging
import re
from dataclasses import dataclass, field as dc_field
from pathlib import Path
from typing import Any

import yaml
from jinja2 import Environment, BaseLoader
from pydantic import BaseModel, ValidationError
from sqlalchemy.orm import Session

from app.core.ai_service import call_ai

logger = logging.getLogger(__name__)


@dataclass
class TaskDefinition:
    task_id: str
    name: str
    description: str
    data_slots: dict[str, str]        # {"primary_entity": "student", "ml_output": "matchmaker"}
    system_prompt_template: str       # Jinja2 template string
    user_prompt_template: str         # Jinja2 template string
    output_schema: dict               # JSON Schema dict — validated at runtime
    jinja2_template: str              # HTML template file name for Jinja2 rendering
    max_tokens: int = 4096
    temperature: float = 0.3
    enable_chat: bool = True
    max_context_tokens: int = 60_000  # hard ceiling on prompt size before truncation


class TaskEngine:
    """
    Load YAML task definitions and execute consultant tasks.
    Data slot resolution maps slot type names to actual data from DB/services.
    """

    # Ordered by priority — module tasks are discovered after platform tasks
    TASK_DIRS: list[Path] = [
        Path("backend/app/platform/tasks"),
        Path("backend/app/modules/school_choice/tasks"),
    ]

    def load_task(self, task_id: str) -> TaskDefinition:
        """Find and parse the first matching task YAML. Module tasks override platform tasks."""
        # Search in reverse order so module tasks take precedence
        for directory in reversed(self.TASK_DIRS):
            candidate = directory / f"{task_id}.yaml"
            if candidate.exists():
                return self._parse_task_yaml(candidate)
        raise ValueError(f"No task definition found for task_id='{task_id}'")

    def _parse_task_yaml(self, path: Path) -> TaskDefinition:
        with open(path) as f:
            raw = yaml.safe_load(f) or {}  # null guard — mirrors yaml_loader.py pattern
        # Missing required keys raise KeyError intentionally — fail loudly at startup
        return TaskDefinition(
            task_id=raw["task_id"],
            name=raw["name"],
            description=raw.get("description", ""),
            data_slots=raw.get("data_slots", {}),
            system_prompt_template=raw["prompts"]["system"],
            user_prompt_template=raw["prompts"]["user"],
            output_schema=raw["output_schema"],
            jinja2_template=raw["jinja2_template"],
            max_tokens=raw.get("max_tokens", 4096),
            temperature=raw.get("temperature", 0.3),
            enable_chat=raw.get("enable_chat", True),
            max_context_tokens=raw.get("max_context_tokens", 60_000),
        )

    def build_messages(
        self,
        task: TaskDefinition,
        entity_id: str,
        db: Session,
    ) -> list[dict[str, str]]:
        """
        Resolve data slots, render Jinja2 templates, check token count, return messages list.
        Raises ValueError if prompt exceeds max_context_tokens after truncation attempt.
        """
        context = self._resolve_data_slots(task.data_slots, entity_id, db)

        env = Environment(loader=BaseLoader())
        system_prompt = env.from_string(task.system_prompt_template).render(**context)
        user_prompt = env.from_string(task.user_prompt_template).render(**context)

        messages = [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ]

        # Token guard — see Section 4 Context Window Strategy for truncation hierarchy
        import litellm as _litellm
        from app.core.ai_service import _build_model_string
        token_count = _litellm.token_counter(model=_build_model_string(), messages=messages)
        if token_count > task.max_context_tokens:
            logger.warning(
                "task=%s entity=%s: prompt is %d tokens, ceiling is %d. Truncating.",
                task.task_id, entity_id, token_count, task.max_context_tokens,
            )
            context = self._truncate_context(context, task)
            system_prompt = env.from_string(task.system_prompt_template).render(**context)
            user_prompt = env.from_string(task.user_prompt_template).render(**context)
            messages = [
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt},
            ]

        return messages

    def execute_task(
        self,
        task_id: str,
        entity_id: str,
        db: Session,
    ) -> dict[str, Any]:
        """
        Full synchronous execution: load task → build messages → call AI → validate output.
        Returns validated dict matching task output_schema.
        Raises on AI failure or schema validation error (caller maps to HTTP 502).
        """
        task = self.load_task(task_id)
        messages = self.build_messages(task, entity_id, db)

        raw_text = call_ai(
            messages,
            max_tokens=task.max_tokens,
            temperature=task.temperature,
        )

        # Strip markdown code fences — some models wrap JSON despite explicit instructions
        raw_text = re.sub(r"^```[a-z]*\n?", "", raw_text.strip())
        raw_text = re.sub(r"\n?```$", "", raw_text.strip())

        try:
            output = json.loads(raw_text)
        except json.JSONDecodeError as exc:
            logger.error("task=%s entity=%s: AI returned invalid JSON: %s", task_id, entity_id, exc)
            raise

        import jsonschema
        jsonschema.validate(instance=output, schema=task.output_schema)

        return output

    def _resolve_data_slots(
        self,
        data_slots: dict[str, str],
        entity_id: str,
        db: Session,
    ) -> dict[str, Any]:
        """
        Map slot type names to actual data objects.
        Add new slot types here — never add resolution logic to the YAML parser.
        """
        resolved: dict[str, Any] = {}
        for slot_name, slot_type in data_slots.items():
            if slot_type == "student":
                resolved[slot_name] = self._load_student(entity_id, db)
            elif slot_type == "matchmaker":
                resolved[slot_name] = self._load_matchmaker_results(entity_id, db)
            else:
                raise ValueError(f"Unknown data slot type: '{slot_type}'")
        return resolved

    def _truncate_context(self, context: dict, task: TaskDefinition) -> dict:
        """
        Reduce context size following the truncation hierarchy in Section 4b.4.
        Modifies a copy — never mutates the original context dict.
        """
        ctx = dict(context)
        if "matchmaker" in ctx and isinstance(ctx["matchmaker"], list):
            results = ctx["matchmaker"]
            # Step 1: reduce to top 5 by final_score
            results = sorted(results, key=lambda r: r.get("final_score", 0), reverse=True)[:5]
            # Step 2: truncate rationale to first sentence
            for r in results:
                rationale = r.get("rationale", "")
                r["rationale"] = rationale.split(".")[0] + "." if "." in rationale else rationale
            # Step 3: remove SHAP explanations (used for badges, not AI reasoning)
            for r in results:
                r.pop("shap_explanation", None)
            ctx["matchmaker"] = results
        return ctx

    def _load_student(self, student_id: str, db: Session) -> dict:
        from app.db.models import Student
        student = db.query(Student).filter(Student.id == student_id).first()
        if not student:
            raise ValueError(f"Student {student_id} not found")
        return {
            "name": student.name,
            "year_of_study": getattr(student, "year_of_study", None),
            "ielts_score": getattr(student, "ielts_score", None),
            "extra_curricular": getattr(student, "extra_curricular", None) or [],
            "awards": getattr(student, "awards", None) or [],
        }

    def _load_matchmaker_results(self, student_id: str, db: Session) -> list:
        from app.modules.school_choice.services.matchmaker_v2 import run_matchmaker
        # run_matchmaker returns list[MatchResult] dataclasses — convert to dicts for Jinja2
        results = run_matchmaker(student_id, db)
        return [vars(r) if hasattr(r, "__dict__") else r for r in results]
```

**Tool Use:** Not applicable. This is a single structured generation call, not a tool-calling agent. The task engine assembles all entity data *before* the AI call — the AI receives a complete context snapshot and returns structured JSON in one pass. No tools, function calling, or mid-generation data retrieval.

**State Management:**

| State | Where Stored | Notes |
|-------|-------------|-------|
| Task definitions | YAML files on disk | Read at request time. No in-memory cache needed at this scale. Add a startup validation step (`validate_all_task_yamls()`) so malformed YAML is caught on deploy, not on user click. |
| Generated plan output | `AcademicPlan` DB row (`html_content`, `recommended_schools`, `action_items`, `version`) | Written after validation passes. Increment `version` on each regeneration. Match the existing pattern in `plan_chat_service.py`. |
| Streaming progress | SSE connection (ephemeral) | No server-side state. Client uses `EventSource` and accumulates chunks. On disconnect, the client reconnects and the task re-executes. Tasks must be idempotent by design — re-running produces an equivalent (not identical) result. |
| Rate limiting | `plan.chat_request_counts` JSONB column | Existing rolling 24-hour window pattern from `plan_chat_service.py`. Apply the same pattern to consultant task execution: 20 calls per entity per user per 24 hours. |

**Context Window Strategy:**

The assembled prompt must fit within the model's context window minus `max_tokens`. Budget:

```
available_context = model_context_window - max_tokens
claude-sonnet-4:   200,000 - 4,096 = ~196,000 tokens
gpt-4o:             128,000 - 4,096 = ~124,000 tokens
gemini-2.5-flash:  1,048,576 - 4,096 = effectively unlimited
```

A typical school choice prompt (student profile + top 10 match results + system instructions) runs 3,000–8,000 tokens — well within all model limits. However, enforce the ceiling to protect against data growth. Default `max_context_tokens: 60_000` in `TaskDefinition` provides a safe buffer for all three models.

Truncation hierarchy (in priority order, applied in `_truncate_context()`):

1. Reduce matchmaker results to top 5 schools (from 10). Remove schools below `fit_score` threshold.
2. Truncate rationale fields in match results to first sentence.
3. Remove `shap_explanation` dicts (used for confidence badges in the rendered HTML, not needed by the AI).
4. Remove low-priority student profile fields (extra_curricular details, award descriptions).
5. If still over budget: raise `ValueError` and do not call the AI. Log at ERROR — this indicates a data growth problem requiring structural remediation, not prompt engineering.

### 4b.1 Structured Outputs with Pydantic

Define the output schema in two places: a JSON Schema in the task YAML (for runtime validation in `execute_task()`), and a Pydantic model in Python (for type-safe downstream use when the task engine integrates with existing services). Both are required.

**Pydantic output model for the school choice plan:**

```python
# backend/app/platform/schemas/consultant_output.py
from __future__ import annotations

from typing import Optional
from pydantic import BaseModel, Field, field_validator


class SchoolRecommendation(BaseModel):
    school_name: str
    jupas_code: Optional[str] = None
    rationale: str = Field(min_length=20)    # guard against one-word rationales
    fit_score: float = Field(ge=0.0, le=1.0)
    confidence_tier: str = Field(pattern=r"^(LOW|MEDIUM|HIGH)$")
    action_items: list[str] = Field(default_factory=list)


class ConsultantPlanOutput(BaseModel):
    student_summary: str = Field(min_length=50)   # guard against empty summaries
    recommended_schools: list[SchoolRecommendation] = Field(min_length=1, max_length=15)
    skill_gaps: list[str] = Field(default_factory=list)
    language_readiness: Optional[str] = None
    action_plan: list[dict] = Field(default_factory=list)

    @field_validator("recommended_schools")
    @classmethod
    def schools_have_rationale(cls, schools: list[SchoolRecommendation]) -> list[SchoolRecommendation]:
        for s in schools:
            if not s.rationale.strip():
                raise ValueError(f"School '{s.school_name}' has empty rationale")
        return schools
```

**How LiteLLM integrates — two approaches:**

Option A: `response_format` with JSON Schema (preferred for OpenAI gpt-4o and Anthropic claude-sonnet-4):

    messages,
    max_tokens=task.max_tokens,
    temperature=task.temperature,

        "type": "json_schema",

            "strict": True,

    },
)
# When response_format is set, the model guarantees valid JSON — skip the markdown fence strip.
# Still validate with Pydantic to catch semantic errors (empty rationale, score out of range).
```

Option B: Prompt-only JSON enforcement (fallback for Gemini and any provider that ignores `response_format`):

Include the complete JSON Schema in the system prompt (see Section 4b.3 prompt template). Strip markdown fences from the response before parsing.

**Validation and retry logic:**

```python
# backend/app/platform/task_engine.py — replace execute_task() with this for production use
import json
import logging
import re
from pydantic import BaseModel, ValidationError

logger = logging.getLogger(__name__)

MAX_RETRIES = 2  # total attempts = 3 (initial + 2 retries)


def _strip_fences(text: str) -> str:
    text = re.sub(r"^```[a-z]*\n?", "", text.strip())
    return re.sub(r"\n?```$", "", text.strip())


def _parse_and_validate(raw_text: str, output_model: type[BaseModel]) -> BaseModel:
    """
    Parse JSON string and validate against Pydantic model.
    Raises ValueError with a diagnostic message on any failure.
    """
    try:
        data = json.loads(_strip_fences(raw_text))
    except json.JSONDecodeError as exc:
        raise ValueError(f"AI returned invalid JSON: {exc}. First 200 chars: {raw_text[:200]}")
    try:
        return output_model(**data)
    except ValidationError as exc:
        raise ValueError(f"AI output failed schema validation: {exc}")


def execute_task_with_retry(
    task: "TaskDefinition",
    messages: list[dict],
    output_model: type[BaseModel],
) -> BaseModel:
    """
    Call AI with retry on validation failure.
    On retry, append the error and correction request to messages so the model can self-correct.
    """
    last_error: Exception | None = None

    for attempt in range(MAX_RETRIES + 1):
        raw_text = call_ai(messages, max_tokens=task.max_tokens, temperature=task.temperature)

        try:
            return _parse_and_validate(raw_text, output_model)
        except ValueError as exc:
            last_error = exc
            logger.warning(
                "task=%s attempt=%d/%d validation failed: %s",
                task.task_id, attempt + 1, MAX_RETRIES + 1, exc,
            )
            if attempt < MAX_RETRIES:
                # Append error context so the model understands what failed
                messages = messages + [
                    {"role": "assistant", "content": raw_text},
                    {
                        "role": "user",
                        "content": (
                            f"Your response failed validation: {exc}\n"
                            "Correct the JSON and respond with ONLY the valid JSON object. "
                            "No explanation, no markdown, no code fences."
                        ),
                    },
                ]

    logger.error(
        "task=%s: all %d attempts failed. Last error: %s",
        task.task_id, MAX_RETRIES + 1, last_error,
    )
    # Surface as HTTP 502 — never expose raw validation errors to non-technical users
    from fastapi import HTTPException
    raise HTTPException(
        status_code=502,
        detail="AI failed to produce a valid plan. Please try again.",
    )
```

Retry surface decision:
- Attempts 1–2 failure: retry silently, log at WARNING.
- Attempt 3 failure: raise `HTTPException(502)` with a user-safe message. Never send raw Pydantic `ValidationError` output to the frontend — it contains internal field names and schema details.

### 4b.2 Async-First Design

**How async works in this stack:**

FastAPI is async-native. `call_ai()` (synchronous, wraps `litellm.completion()`) is safe to call from a regular `def` route handler or a background task. It blocks the OS thread but not other FastAPI workers if the deployment uses multiple uvicorn workers.

`call_ai_stream()` (async generator, wraps `litellm.acompletion()`) must be called from an `async def` route and returned as a `StreamingResponse`. It is non-blocking — the event loop handles other requests while waiting for AI tokens.

**The one critical mistake — blocking the event loop with synchronous streaming:**

```python
# WRONG — using litellm.completion(stream=True) inside an async route
@router.post("/consultant/tasks/{task_id}/stream")
async def stream_task(task_id: str):
    # litellm.completion(stream=True) is synchronous I/O.
    # Calling it here blocks the ENTIRE FastAPI event loop for the duration of the stream.
    # Every other user request queues behind this one 10-30s AI call.
    response = litellm.completion(model=..., messages=..., stream=True)  # BLOCKS EVENT LOOP
    for chunk in response:
        yield f"data: {chunk}\n\n"
```

```python
# CORRECT — use litellm.acompletion(stream=True) via call_ai_stream()
@router.post("/consultant/tasks/{task_id}/stream")
async def stream_task(task_id: str, ...):
    # call_ai_stream() uses litellm.acompletion internally — yields control to the event loop
    # between each token, allowing other requests to be served concurrently.
    return StreamingResponse(
        call_ai_stream(messages, max_tokens=4096, temperature=0.3),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )
```

**Stream vs. await — complete decision table:**

| Use Case | Pattern | Reason |
|----------|---------|--------|
| User-facing "Generate Plan" button | `call_ai_stream()` → `StreamingResponse` | First token visible in ~1s. Users abandon after 3s of silence. |
| Background task storing plan to DB | `call_ai()` synchronous | Need complete JSON before Pydantic validation. Cannot validate a partial stream. |
| Plan chat edit (`plan_chat_service.py` pattern) | `call_ai()` synchronous | JSON patch must be complete before `_apply_patch()` runs. Same reason. |
| Admin bulk generation in a background worker | `call_ai()` in `asyncio.gather()` calls if parallelism needed | Batch jobs care about throughput, not streaming UX. |

**Frontend `EventSource` reference (for the frontend engineer):**

```javascript
// frontend — EventSource reconnects automatically on network drop
const source = new EventSource(
  `/api/v1/consultant/tasks/${taskId}/stream?entity_id=${entityId}`
);
source.onmessage = (event) => {
  appendToken(event.data);  // append each token to the display buffer
};
source.addEventListener("done", () => {
  source.close();
  onGenerationComplete();  // trigger Pydantic-validated save on the backend
});
source.onerror = () => {
  source.close();
  showError("Generation was interrupted. Please try again.");
};
```

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation — always enforced:**

```
System prompt: role definition + output contract + format instructions + grounding constraints
User prompt:   entity data (the variable "input") + the specific task instruction
```

Never mix entity data into the system prompt. The system prompt is identical for every execution of a given task and is eligible for Anthropic prompt caching (which caches the system prompt prefix across calls with the same system content, reducing latency and cost).

**System prompt template for the school choice academic plan task (in task YAML):**

```yaml
prompts:
  system: |
    You are a professional academic consultant generating a structured university application plan
    for a student.

    GROUNDING RULES — these are absolute constraints, not suggestions:
    1. Only reference schools, programs, JUPAS codes, and admission scores that appear in the
       provided match results. Do not use your training knowledge about any school's programs,
       class sizes, or reputation — only what is in the data.
    2. Every recommended school must have a rationale of at least 2 sentences, each sentence
       grounded in a specific field from the student profile (e.g., "Your IELTS score of 7.0
       meets the language requirement of 6.5.").
    3. Respond with ONLY a valid JSON object matching the schema below. No markdown, no
       explanation, no code fences. The response must begin with '{' and end with '}'.

    OUTPUT SCHEMA:
    {
      "student_summary": "string — minimum 50 characters, student name and key profile facts",
      "recommended_schools": [
        {
          "school_name": "string — exact name from match results",
          "jupas_code": "string or null",
          "rationale": "string — minimum 20 characters, data-grounded",
          "fit_score": "float 0.0 to 1.0",
          "confidence_tier": "LOW | MEDIUM | HIGH",
          "action_items": ["string"]
        }
      ],
      "skill_gaps": ["string"],
      "language_readiness": "string or null",
      "action_plan": [
        {"task": "string", "priority": "High | Medium | Low", "deadline": "string or null"}
      ]
    }

  user: |
    Student Profile:
    {{ student | tojson(indent=2) }}

    School Match Results (ranked by final_score, highest first):
    {{ matchmaker | tojson(indent=2) }}

    Generate the academic plan now.
```

**Few-shot examples:** Do not embed static few-shot examples inline for this use case. Entity data is multi-KB; adding inline examples pushes the prompt toward the token ceiling. If rationale quality degrades in production, add one compact example in the system prompt using clearly fictional placeholder data (e.g., `"school_name": "EXAMPLE UNIVERSITY"`) that cannot be confused with real entities.

**Token budget — always set `max_tokens` explicitly in task YAML:**

Never leave `max_tokens` unset. LiteLLM does not impose a default ceiling — an unconstrained call can generate tens of thousands of tokens and cost 10–50x the expected amount if the model decides to be verbose. A full school choice plan with 10 schools and all sections fits comfortably within 2,000 output tokens; 4,096 is a conservative ceiling.

### 4b.4 Context Window Management

**Strategy for this system type (single-call content generation):**

The risk is not conversation history accumulation (this is a single-call pattern) — it is entity data size growth as the school database and student profile fields expand over time. Manage this proactively.

**Measure before every call:**

```python
# In task_engine.py build_messages() — always measure after building messages
import litellm as _litellm
from app.core.ai_service import _build_model_string

token_count = _litellm.token_counter(model=_build_model_string(), messages=messages)

if token_count > task.max_context_tokens:
    logger.warning(
        "task=%s entity=%s: prompt is %d tokens (ceiling: %d). Applying truncation.",
        task.task_id, entity_id, token_count, task.max_context_tokens,
    )
    context = self._truncate_context(context, task)
    # Rebuild messages and re-measure
```

**Truncation hierarchy for school choice plans (apply in order, stop when under ceiling):**

1. Reduce matchmaker results to top 5 by `final_score` (from 10). Remove schools below a `fit_score` of 0.3.
2. Truncate each `rationale` string to the first sentence.
3. Remove `shap_explanation` dicts from all match results (these are used for the rendered confidence badge tooltip, not for AI reasoning).
4. Remove `extra_curricular` details and `awards` descriptions from the student profile (keep counts only).
5. If still over `max_context_tokens` after all four steps: raise `ValueError` and do not call the AI. Log at ERROR with the token count and entity ID — this is a data growth signal requiring structural remediation.

**For plan chat (existing `plan_chat_service.py` pattern):**

The chat service already implements the correct pattern: it passes only the compact `recommended_schools` and `action_items` JSON as context, not the full rendered HTML. This keeps chat context under 2K tokens. Maintain this exact pattern when generalising `plan_chat_service.py` to work with any task output — always pass the task's structured JSON output as context, never the rendered Jinja2 HTML.

### 4b.5 Cost and Latency Budget

**Per-call cost estimate (school choice plan, typical input):**

| Model | Input tokens (est.) | Output tokens (est.) | Cost/call (USD) | At 500 plans/day | At 5,000 plans/day |
|-------|--------------------|--------------------|----------------|-----------------|-------------------|
| `claude-sonnet-4-20250514` | ~6,000 | ~2,000 | ~$0.054 | ~$27/day | ~$270/day |
| `gpt-4o` | ~6,000 | ~2,000 | ~$0.074 | ~$37/day | ~$370/day |
| `gemini-2.5-flash` | ~6,000 | ~2,000 | ~$0.005 | ~$2.50/day | ~$25/day |

Validate these estimates against real `response.usage` data after launch. LiteLLM populates `response.usage.prompt_tokens` and `response.usage.completion_tokens` consistently across all three providers — log these fields from `call_ai()` from day one.

**Exact-match caching (Redis — implement before launch):**

Cache the final validated plan output by a hash of the input context. Serve cached output when the entity data has not changed since last generation. This eliminates redundant AI calls for the most common pattern: a counselor generates a plan, shows it to a parent, then the parent opens the same plan again.

```python
# Cache key construction — in task_engine.py execute_task()
import hashlib, json

def _plan_cache_key(task_id: str, entity_id: str, context: dict) -> str:
    """Deterministic cache key based on task + entity + input data content."""
    context_hash = hashlib.sha256(
        json.dumps(context, sort_keys=True, ensure_ascii=False).encode()
    ).hexdigest()[:16]
    return f"consultant_plan:{task_id}:{entity_id}:{context_hash}"
```

Cache TTL: 24 hours. Invalidate on any update to the student record or matchmaker results for that entity. Do not use semantic caching for this use case — school choice plans are entity-specific; a semantically similar student should produce a different plan, not a cached result from another student.

**Sub-task model routing — use cheaper models for non-user-visible tasks:**

| Sub-task | Recommended approach | Reason |
|----------|---------------------|--------|
| Full plan generation | `claude-sonnet-4` or `gpt-4o` (use `AI_PROVIDER` env) | Quality is user-visible; plan content is the primary product |
| Plan section chat edit | `gemini-2.5-flash` (override in chat task YAML) | Small JSON patch; quality difference is not perceptible; 10x cheaper |
| Confidence tier badge | Python rule — `compute_data_completeness()` thresholds | Never use AI for deterministic computations already in `matchmaker_v2.py` |
| Task routing (which YAML to load) | URL parameter / request body — no AI | Never use AI for deterministic routing decisions |

**Latency budget:**

- Streaming plan generation: first token visible to user within 2s; full plan complete within 30s. If first token exceeds 5s, investigate: model under load, overly long prompt, or cold-start latency from provider.
- Sync plan chat edit: full round-trip (AI call + DB write + HTML regeneration) within 8s. The existing `plan_chat_service.py` achieves this comfortably.
- If generation consistently exceeds 30s: profile `token_count` in the logs first — the most common cause is a prompt that grew larger than expected, not model speed. Switching to a faster model without fixing the prompt size is a recurring mistake.

## 5. Evaluation Strategy

### Dimensions

Each dimension below is drawn directly from the domain rubric ingredients in Section 1b. Generic AI jargon has been replaced with school-choice-specific pass/fail criteria.

| Dimension | PASS | FAIL | Measurement | Priority |
|-----------|------|------|-------------|----------|
| **Output schema validity** | Response parses as valid JSON; all required fields (`student_summary`, `recommended_schools`, `action_plan`) are present and non-empty; `fit_score` is 0.0–1.0; `confidence_tier` is one of LOW/MEDIUM/HIGH. | `json.JSONDecodeError` raised, required field absent, `fit_score` outside range, or `confidence_tier` is an unexpected string. | Code — `jsonschema.validate()` + Pydantic `ConsultantPlanOutput` | Critical |
| **Eligibility accuracy** | Every school in `recommended_schools` is one the student's data qualifies them to apply to: grade band matches, lottery window is open (if present in data), language program eligibility conditions are met. No school appears that the student's record disqualifies. | Plan includes a school the student's record explicitly disqualifies (e.g., JUPAS minimum score above student score, closed application window recorded in school data). | LLM Judge — scored against the grounding rules in the task YAML system prompt; calibrated against a school counselor-labeled reference set before trusting | Critical |
| **Data grounding — no fabricated program details** | Every named program, language track, or school feature cited in any `rationale` field exists as a field in the school's entity record passed to the system. No description draws on the model's training knowledge of a school. | A rationale mentions "IB programme", "dual-language track", "small class sizes", or any other specific program/feature not present in the matchmaker results JSON provided to that call. | LLM Judge — compares each rationale sentence against the verbatim matchmaker results JSON for that call; flags any claim without a matching source field | Critical |
| **Confidence badge accuracy** | Each school's `confidence_tier` in the AI output matches what `compute_data_completeness()` would assign for that school's record completeness. HIGH is only present when all eligibility-relevant fields are populated. | AI assigns HIGH to a school whose record is missing one or more eligibility-determining fields (e.g., no language assessment score for a language program school). | Code — compare AI-returned `confidence_tier` per school against `compute_data_completeness()` output for the same school record; assert agreement within one tier | High |
| **Student-specificity** | `student_summary` contains the student's name, at least one referenced field value (e.g., IELTS score, year of study), and school rankings that match the `final_score` order from matchmaker results. No sentence in the summary is universally true for all students. | `student_summary` omits the student's name, contains no field-value references, or the ranking order of schools diverges from matchmaker `final_score` ordering without a stated reason. | LLM Judge — rubric: "Does this summary contain at least 2 student-specific data references? Could this exact text appear unchanged in a plan for a different student?" Score 1–5; PASS >= 4; calibrate against education consultant labels | High |
| **Action item sequence logic** | `action_plan` items are ordered chronologically by deadline (earliest deadline first) when deadline fields are present in school data. Items distinguish family-controlled actions from contingent actions. No deadline value appears that is absent from the school entity records. | Action items list a deadline not present in any school record, or items requiring completion before a lottery registration are listed after the lottery registration step. | LLM Judge — checks deadline values against school records JSON; checks that prerequisite steps precede dependent steps (e.g., "register for language assessment" before "submit lottery application") | Medium |
| **SSE delivery completeness** | The streaming response delivers a `done` event; accumulated token buffer assembles into valid JSON after the stream closes; no partial or truncated JSON is written to the database. | Client receives an SSE `error` event mid-stream, or the accumulated buffer fails `json.loads()` after the `done` event, or the DB row contains an incomplete plan. | Code — integration test using `httpx.AsyncClient` with SSE; assert `done` event received and buffer is valid JSON | Critical |
| **FERPA data minimization** | Student PII fields sent to the AI provider are limited to fields declared in the task's `data_slots`; no fields outside those slots appear in the prompt. The provider receives no fields unnecessary for generating the plan. | Student record fields not listed in `data_slots` (e.g., medical notes, disciplinary records, sibling data) appear in the rendered user prompt sent to the provider. | Code — inspect rendered prompt string for presence of field names outside `data_slots` declaration; assert absence of known PII field names not in the slot definition | High |

    provider.add_span_processor(BatchSpanProcessor(exporter))
    trace.set_tracer_provider(provider)

_tracer = trace.get_tracer("consultant_engine")

# Wrap call_ai() with a span — add at the start of the function body
def call_ai(messages: list[dict], **kwargs) -> str:
    with _tracer.start_as_current_span("call_ai") as span:
        span.set_attribute("task_id", kwargs.get("task_id", "unknown"))
        span.set_attribute("input_token_count", kwargs.get("_token_count", 0))
        # ... existing call_ai body ...
        span.set_attribute("output_length", len(result))
        return result

### Reference Dataset

**Size:** 20 examples minimum before production deployment. Start building during implementation, not after.

**Composition:**

| Scenario Type | Count | Purpose |
|---------------|-------|---------|
| Complete student record, 10 schools with full data | 4 | Critical path — ideal conditions; establishes quality baseline |
| Sparse student record (missing language assessment, no GPA) | 4 | Known failure mode: eligibility blind spots; confidence inflation |
| Student with one clearly ineligible school in match results | 3 | Eligibility accuracy — verifies the system correctly excludes or flags |
| Student record where all fields populated but key eligibility field absent from entity schema | 2 | Confidence score inflation from partial records — the hardest failure mode to catch |
| Long prompt triggering truncation (15+ schools in matchmaker results) | 2 | SSE delivery + truncation correctness |
| Student with identical generic profile (minimal data, no distinguishing fields) | 2 | Boilerplate drift — the plan must still reference what data exists |
| School with fabrication risk (well-known school with extensive training data) | 3 | Fabricated program details — LLM most likely to hallucinate for schools it "knows" |

**Labeling:**

- Code-based checks (schema validity, field presence, score range): automated — no labeling required.
- Eligibility accuracy, data grounding, student-specificity, action item logic: labeled by a practicing K-12 school counselor using the Section 1b rubric criteria. Each plan section receives PASS / FAIL / NEEDS REVIEW per rubric dimension. Target 2 independent counselor reviews per example to establish inter-rater agreement.
- LLM judge calibration: run the LLM judge against all 20 human-labeled examples; compute correlation. Do not use the LLM judge in CI/CD until correlation with human labels is ≥ 0.70 on the eligibility accuracy and data grounding dimensions.
- Timeline: begin dataset construction during sprint 1 of Phase 5 implementation; complete labeling before first staging deployment.

## 6. Guardrails

### Online (Real-Time)

These run on every plan generation request before the output is written to the database. Each adds latency — keep them fast.

| Guardrail | Trigger | Intervention | Implementation | Latency Impact |
|-----------|---------|--------------|----------------|----------------|
| **Schema validation failure** | `jsonschema.validate()` or Pydantic raises on AI output | Retry (up to 2 times per `execute_task_with_retry()`); on third failure, return HTTP 502 with user-safe message — never show raw JSON error | Already implemented in Section 4b.1 retry logic | < 1ms (synchronous Python) |
| **Confidence tier mismatch** | AI-returned `confidence_tier` for a school is higher than the tier `compute_data_completeness()` assigns for that school's actual record | Override AI confidence with the code-computed tier before writing to DB; log discrepancy at WARNING with task_id and school_name | Code check in `task_engine.execute_task()` after validation; compare AI tier against `compute_data_completeness()` result; downgrade if AI tier is higher | < 5ms (Python dict lookup) |
| **Prompt PII scope violation** | Rendered user prompt string contains a field name known to be outside the task's `data_slots` declaration (checked against a blocklist of sensitive field names: `medical_notes`, `disciplinary_record`, `sibling_data`, `ssn`, `dob`) | Abort the AI call; log at ERROR with task_id and entity_id; return HTTP 400 — do not proceed with a prompt containing out-of-scope PII | Code check in `task_engine.build_messages()` before calling `call_ai()`; regex scan of rendered prompt against blocklist | < 2ms |
| **SSE stream termination without done event** | `EventSource.onerror` fires or connection closes before the `done` event is sent | Frontend shows "Generation was interrupted. Please try again." — no partial plan is saved to DB; backend logs at WARNING with entity_id | Frontend `EventSource.onerror` handler (see Section 4b.2); backend does not write to DB until full response is validated | Client-side only |

### Offline (Flywheel)

These run on a sampled batch of completed generations, feeding the quality improvement loop. They do not block user-facing requests.

| Metric | Sampling Strategy | Measurement | Action on Degradation |
|--------|------------------|-------------|----------------------|
| **Data grounding rate** — percentage of plans where all rationale claims are traceable to input data | 10% random sample of all generated plans; 100% of plans flagged by counselors or users | LLM Judge against the rationale-grounding rubric from Section 1b | If grounding rate drops below 85% in any 7-day window: review system prompt for drift; check if new school records contain fields with ambiguous names that increase hallucination risk |
| **Student-specificity score** | 10% random sample | LLM Judge with the specificity rubric from Section 1b (score 1–5; PASS >= 4) | If average score drops below 3.5 in any 7-day window: inspect whether student records have become sparser (fewer populated fields) or whether prompt truncation is removing key profile data |
| **Confidence tier override rate** — percentage of plans where the online guardrail downgraded a HIGH AI confidence to MEDIUM or LOW | 100% (logged automatically by the guardrail) | Code — count of downgrades / total generations | If override rate exceeds 15% in any 7-day window: the AI is systematically overconfident; adjust the system prompt grounding rules to reinforce confidence tier assignment criteria |
| **Validation retry rate** — percentage of calls that required 1 or more retries to produce valid JSON | 100% (logged automatically by `execute_task_with_retry()`) | Code — count of `attempt > 0` completions / total completions | If retry rate exceeds 10% in any 7-day window: log temperature and prompt token count for failing calls; consider lowering temperature from 0.3 to 0.1 or adding a JSON-enforcing example to the system prompt |
| **Action item sequence errors** | 20% random sample of plans with 3+ action items | LLM Judge against the sequence logic rubric from Section 1b | If error rate exceeds 20% in any monthly review: add a sequence-ordering instruction to the task YAML system prompt; validate against reference dataset before deploying |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix (self-hosted, open-source)

No existing tracing infrastructure was detected in the project. Arize Phoenix is the default — it is self-hostable, requires no external platform account, and instruments via OpenTelemetry. Run Phoenix as a local process alongside uvicorn in development; in production, run it as a separate long-lived process on the same host or as a sidecar.

```bash
# Development: launch Phoenix before starting uvicorn
pip install arize-phoenix
python -m phoenix.server.main serve
# UI available at http://localhost:6006

# Production: run as a background service
nohup python -m phoenix.server.main serve --port 6006 > phoenix.log 2>&1 &
```

**Key Metrics to Track:**

| Metric | Source | Why It Matters |
|--------|--------|----------------|
| `call_ai` span duration (p50, p95) | Phoenix span attribute | Baseline for latency budget (target p95 < 30s). Spike = model under load or prompt grew. |
| `prompt_tokens` + `completion_tokens` per call | `response.usage` logged in `call_ai()` | Cost tracking and prompt growth detection. Alert if `prompt_tokens` exceeds 10,000 — indicates truncation is not firing or data growth needs attention. |
| Schema validation retry count (0, 1, 2, 3+) | Application log via Phoenix span attribute | Structural AI reliability signal. Retry rate above 10% is a prompt engineering or temperature problem. |
| Confidence tier override rate | Application log in `execute_task()` guardrail | AI overconfidence signal. Rate above 15% means the system is presenting families with inflated confidence. |
| Generation error rate (HTTP 502 responses from consultant endpoints) | FastAPI access log / Phoenix span status | User-facing failure rate. Alert immediately if above 2% in any 1-hour window. |

**Alert Thresholds:**

| Condition | Threshold | Action |
|-----------|-----------|--------|
| Generation error rate (HTTP 502) | > 2% of requests in any 1-hour window | Immediate: check AI provider status page; check `call_ai()` error logs for auth failures or rate limits |
| p95 generation latency | > 45s sustained over 30 minutes | Investigate: log `prompt_tokens` for slow calls; check if truncation is not applying; check provider status |
| Schema validation retry rate | > 10% in any 7-day rolling window | Within 48 hours: review system prompt; consider lowering temperature; run reference dataset against updated prompt |
| Confidence tier override rate | > 15% in any 7-day rolling window | Within 48 hours: review confidence tier assignment language in system prompt |
| `prompt_tokens` per call | > 10,000 tokens on any single call | Immediate: investigate why truncation did not fire; check `_truncate_context()` logic and `max_context_tokens` config |

**Smart Sampling Strategy:**

Weight production interactions toward higher-risk signals when selecting plans for human (counselor) review. Sample at the rates below from the total plan generation volume:

| Signal | Sample Rate | Rationale |
|--------|------------|-----------|
| Plans where confidence tier override fired (AI overconfident) | 100% | Highest data quality risk — families may have acted on inflated confidence |
| Plans where validation retried at least once | 50% | Structural instability signal — even if eventually successful, output quality may be degraded |
| Plans for students with sparse records (< 4 populated eligibility fields) | 25% | Highest eligibility blind spot risk; sparse data is the primary failure mode trigger |
| Plans for schools with well-known national reputations (top-10 school records by external ranking) | 20% | Highest fabrication risk — model training data is richest for these schools |
| Baseline random sample from all other plans | 5% | General quality monitoring |

Review cadence: platform operator reviews sampled plans monthly using the Section 1b rubric dimensions. School counselor reviews a subset of flagged plans (confidence override, sparse records) within 2 weeks of generation. Findings feed directly into the offline flywheel actions in Section 6.
