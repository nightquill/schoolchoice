> Visual and interaction contract for Phase 6: Deployment and Production Readiness.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase UI Scope

Phase 6 is primarily backend/DevOps (deployment configs, CI/CD, secrets, seed scripts). The **single new UI surface** is the admin user management page (SEC-01, SEC-02). Additionally, route guards must hide admin-only nav links and return 403 for non-admin API calls.

**New UI surfaces:**
1. Settings page with Users tab (admin-only)
2. Nav link visibility based on role (hide "Users" / "Settings" for non-admin)
3. Route guard redirecting non-admin users away from `/settings`

**No new UI surfaces for:** deployment, CI/CD, secrets, seed scripts (all backend/script deliverables).

## Component Inventory

### 1. SettingsPage (new route: `/settings`)

**Purpose:** Container page for admin-only settings. Uses shadcn `<Tabs>` for future extensibility (only "Users" tab in Phase 6).

**Layout:**
- Page title: "Settings" (`--font-size-2xl`, `--font-weight-bold`)
- `<Tabs>` with single tab: "Users"
- Content area: `<Card>` wrapping the user management table
- Padding: `--space-6` on all sides

**Route guard:** `App.jsx` checks `user.role === 'admin'` before rendering this route. Non-admin users hitting `/settings` are redirected to `/dashboard`. Nav link "Settings" (with `Settings` lucide icon, 18px) only renders for admin users.

**Mobile (<768px):** Full-width, no layout changes needed (single-column content).

### 2. UserManagementTable (inside Settings > Users tab)

**Purpose:** CRUD table for admin to view, create, edit, and delete user accounts.

**Anatomy:**
- Header row: "Create User" button (primary, accent fill) aligned right above table
- shadcn `<Table>` with columns: Name, Email, Role, Created, Actions
- Name column: `--font-size-md`, `--font-weight-bold`
- Email column: `--font-size-md`, `--color-text-secondary`
- Role column: `<Badge>` with role-specific colors (see color section)
- Created column: `--font-size-sm`, `--color-text-secondary`, formatted as "Apr 28, 2026"
- Actions column: `<DropdownMenu>` trigger (vertical dots icon, `MoreVertical` lucide, 16px) with items: "Edit", "Delete"

**Empty state:** Rendered inside the table card when no users exist (unlikely in practice, since the current admin user always exists).

**Row hover:** `background: var(--color-background)` on hover (matches existing entity list table pattern).

### 3. CreateEditUserDialog (shadcn `<Dialog>`)

**Purpose:** Modal dialog for creating a new user or editing an existing user.

**Anatomy:**
- Title: "Create User" or "Edit User" (`--font-size-xl`, `--font-weight-bold`)
- Form fields (stacked, `--space-4` gap):
  - Full Name: shadcn `<Input>`, required
  - Email: shadcn `<Input>` type="email", required
  - Password: shadcn `<Input>` type="password", required for create, optional for edit (placeholder: "Leave blank to keep current password")
  - Role: shadcn `<Select>` with options: "Admin", "Counsellor"
- Footer: "Cancel" (ghost button) + "Save" (primary button, accent fill)
- Width: 480px max on desktop, full-width on mobile

**Validation:**
- Client-side: required fields highlighted with `--color-error` border on submit if empty
- Server-side: duplicate email returns error toast "A user with this email already exists."

### 4. DeleteUserConfirmation (shadcn `<Dialog>`)

**Purpose:** Confirmation dialog before deleting a user account.

**Anatomy:**
- Title: "Delete User" (`--font-size-xl`, `--font-weight-bold`)
- Body: "Are you sure you want to delete {user name}? This action cannot be undone."
- Footer: "Cancel" (ghost button) + "Delete" (destructive button, `--color-error` fill, white text)

**Self-delete prevention:** If the user being deleted is the current logged-in admin, the "Delete" action in the dropdown is disabled with tooltip: "You cannot delete your own account."

## Interaction Contracts

### Create User Flow

1. Admin clicks "Create User" button above table
2. CreateEditUserDialog opens with empty form
3. Admin fills Name, Email, Password, selects Role
4. Admin clicks "Save" -> button enters loading state (spinner, disabled)
5. On success: dialog closes, table refreshes, toast "User created successfully."
6. On error (duplicate email): toast "A user with this email already exists." Dialog stays open.
7. On error (server): toast "Failed to create user. Please try again."

### Edit User Flow

1. Admin clicks vertical dots on a user row -> dropdown opens
2. Admin clicks "Edit" -> CreateEditUserDialog opens pre-populated with user data
3. Password field is empty with placeholder "Leave blank to keep current password"
4. Admin modifies fields, clicks "Save"
5. On success: dialog closes, table refreshes, toast "User updated successfully."
6. On error: toast "Failed to update user. Please try again."

### Delete User Flow

1. Admin clicks vertical dots on a user row -> dropdown opens
2. Admin clicks "Delete" -> DeleteUserConfirmation dialog opens
3. Admin clicks "Delete" (destructive) -> button enters loading state
4. On success: dialog closes, user removed from table, toast "User deleted."
5. On error: toast "Failed to delete user. Please try again."
6. Self-delete: "Delete" dropdown item is disabled; tooltip explains why

### Route Guard Behavior

1. Non-admin user navigates to `/settings` (via URL bar or any other means)
2. App redirects to `/dashboard` immediately
3. No error message shown (route simply does not exist for non-admin users)
4. Nav sidebar: "Settings" link not rendered for non-admin users

### API 403 Handling

1. If a non-admin user somehow calls an admin-only API endpoint
2. Backend returns 403 with JSON `{"detail": "Admin access required"}`
3. Frontend displays toast: "You do not have permission to perform this action."

## Accessibility Requirements

- All interactive elements have visible focus ring using `--color-primary` (`box-shadow: 0 0 0 2px #2563EB`)
- Delete confirmation dialog traps focus inside dialog when open
- DropdownMenu items keyboard-navigable (arrow keys, Enter to select, Escape to close)
- Role `<Select>` accessible with keyboard
- Self-delete disabled state communicated via `aria-disabled="true"` and tooltip accessible on focus
- Mobile touch targets: minimum 44px height for all buttons (WCAG 2.5.5)
- Table uses `<thead>` with `scope="col"` on column headers for screen readers
