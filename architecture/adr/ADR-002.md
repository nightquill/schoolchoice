# ADR-002: Rule-Based Matching over Machine Learning
# Intelligent Academic Advisor — MVP
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-008, REQ-009, REQ-002, REQ-007, REQ-016, REQ-017, REQ-018, REQ-019, REQ-020

---

## Context

The system's core function is to match students to schools and produce a ranked recommendation list with explanations (REQ-016 through REQ-020). A matching algorithm must be chosen. Two broad approaches were evaluated:

**Option A — Machine Learning (ML) model:**
Train a model (e.g., logistic regression, gradient boosting, or a neural network) on historical student-school placement data. The model produces a match score as a continuous output. Explanations could be approximated via feature importance or SHAP values.

**Option B — Rule-based scoring:**
Define explicit criteria and fixed weights. Apply a deterministic pipeline: (1) filter schools that fail minimum grade thresholds, (2) compute a weighted score from three factors (grade match, interest alignment, school-strengths alignment), (3) rank and return the top 5. The factors and weights are constants defined in backend source code.

---

## Decision

**Rule-based scoring (Option B) is adopted for MVP.**

---

## Reasoning

1. **Explicit requirement.** REQ-008 mandates that no machine learning models may be introduced in MVP. This decision is non-negotiable for the current build.

2. **Transparency and interpretability.** REQ-009 requires that system outputs be transparent and interpretable; black-box outputs are not acceptable. Rule-based scoring produces a directly explainable result: the explanation field in each recommendation can enumerate which factors contributed, what weight each carries, and what score was computed. An ML model's output cannot provide this level of explanation without post-hoc approximation methods that introduce their own uncertainty.

3. **Decision-support principle.** The system is decision-support, not automation (REQ-002). The counselor must be able to understand and challenge any recommendation the system produces. Rule-based logic enables this: a counselor can trace the exact path from student data to recommendation. An ML model produces a score that a non-technical counselor cannot interrogate.

4. **No training data is available.** An ML model requires labelled historical data (prior student placements and outcomes). No such dataset exists at MVP stage. Rule-based logic requires only domain knowledge, which counselors can directly supply and validate.

5. **Build simplicity.** A rule-based engine is implemented entirely within the FastAPI backend as a pure Python module with no external dependencies beyond standard arithmetic. There is no training pipeline, no model artefact management, no inference infrastructure, and no versioning of model weights.

6. **Extensibility.** REQ-007 requires the architecture to support future addition of ML-based scoring without rewriting MVP components. The matching engine is isolated as an internal module behind a clean interface (receives student data and school list; returns ranked recommendation list). An ML-based module can replace this interface in a future iteration without altering the API contracts, database schema, or frontend.

---

## Consequences

**Accepted trade-offs:**

- **Static weights.** Scoring weights for the three factors (grade match, interest alignment, school-strengths alignment) are fixed constants in MVP (REQ-018). A counselor cannot tune them. This may produce recommendations that do not fully reflect a particular counselor's priorities for a specific student. The counselor must use their professional judgment to interpret and override recommendations.
- **No learning from outcomes.** The rule-based engine cannot improve based on actual student placements. Its quality is bounded by the accuracy of the rules as initially defined.
- **Manual rule maintenance.** If the scoring criteria need to change (e.g., new factors, different weights), a code change and redeployment are required. There is no admin interface for rule configuration in MVP.

**Not affected:**
- The recommendation output schema (school name, score, explanation, gaps) remains valid regardless of the computation method. A future ML module would produce the same output fields.
- The API contract for the recommendations endpoint does not change.
