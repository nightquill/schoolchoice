# ADR-007: HTML Plan Document (Server-Rendered HTML over PDF Generation)
# Intelligent Academic Advisor — v2 Pipeline
# Date: 2026-03-27
# Status: Accepted

---

## REQ-IDs

REQ-077, REQ-078, REQ-096

---

## Context

The v2 AcademicPlan must produce a rich, structured document presenting a student's
full academic profile, recommended schools with gap analysis, an action plan timeline,
and supporting appendices (REQ-077). This document must be:

- Viewable in the browser (REQ-096).
- Printable (REQ-077 §9.2).
- Stored in the database as the persistent plan record (AcademicPlan.html_content).
- Self-contained: no external dependencies at render time.
- Generated entirely on the backend with no JavaScript in the output.

Three document format options were evaluated:

---

## Options Evaluated

### Option A — PDF generation (WeasyPrint or ReportLab)
The backend generates a PDF file, stores it to disk, and returns a download URL.
The frontend links to the PDF for view and print.

**Rejected because:**
- **Dependency weight.** WeasyPrint requires GTK/Cairo native libraries; ReportLab
  has its own layout engine. Both significantly increase the Docker image size and
  introduce native dependency management.
- **Non-inspectable.** PDFs are binary blobs. The stored plan cannot be queried,
  diffed, or updated without re-generating the entire document. Storing a binary
  file in the database or on disk is more operationally complex than storing text.
- **Browser rendering variability.** PDF viewers embedded in browsers (Chrome,
  Firefox, Safari) render differently and do not always support inline display
  without a download prompt. Mobile PDF rendering is inconsistent (REQ-052).
- **Print parity.** A web page with `@media print` CSS produces print output that
  is functionally equivalent to a PDF for the use cases here (A4, clean layout,
  no interactive elements). This eliminates the need for a separate format.

### Option B — Markdown + frontend rendering (react-markdown)
The backend generates Markdown text; the frontend renders it to HTML using a
Markdown library.

**Rejected because:**
- Markdown has limited layout capability. Tables, multi-column gap analysis grids,
  and styled status badges require HTML/CSS constructs that Markdown cannot express
  cleanly without raw HTML blocks.
- The rendering library (react-markdown + plugins) would need to be configured and
  maintained in the frontend. Styling the rendered output to match the application
  design requires additional CSS handling.
- The document is a one-way output (not interactive), so client-side rendering adds
  no benefit over server-rendered HTML.

### Option C — Server-rendered HTML with inline CSS
The backend builds a complete HTML string using Python string templates or a
templating engine (e.g., Jinja2). The HTML uses only inline CSS and an embedded
`<style>` block (no external stylesheet references). The completed document is stored
as text in `AcademicPlan.html_content` and served directly by the GET /plan endpoint.
A `@media print` block is included in the embedded styles.

**Selected — see Decision below.**

---

## Decision

**Server-rendered HTML with inline CSS is adopted for the AcademicPlan document.**

**Format constraints (REQ-077):**
- All CSS is inline or within a `<style>` block in the document `<head>`.
  No external stylesheet `<link>` elements.
- A `@media print` block in the embedded `<style>` is mandatory.
- No `<script>` tags or inline JavaScript of any kind.
- No external image references; any icons or decorative elements use CSS or Unicode
  symbols only.
- The document must render correctly when its `html_content` string is placed in
  an `<iframe srcdoc="...">` or a standalone browser tab (REQ-096).

**Backend implementation:**
- Use Jinja2 templates (already a FastAPI/Python ecosystem standard) to compose
  the 7 sections.
- The template receives a structured data context (student profile, school targets,
  action items) assembled by the plan generator module before rendering.
- The rendered string is stored in `AcademicPlan.html_content` (TEXT column).

**Frontend rendering (REQ-096):**
- The frontend fetches `html_content` from GET /plan and renders it in an
  `<iframe srcdoc={html_content}>` within the Academic Plan page route.
- Print is triggered via the browser's native print dialog (`window.print()` from
  within the iframe context or a parent print button that focuses the iframe).
- The `@media print` styles in the document itself control the printed layout;
  no frontend print CSS is needed.

---

## Reasoning

1. **Zero new dependencies.** Jinja2 is already part of the FastAPI ecosystem.
   No native library compilation, no added Docker layer. The backend image size is
   unchanged.

2. **Inspectable and diffable.** HTML text stored in a database column can be
   indexed, queried, and version-controlled. Plan versions are tracked via the
   `version` integer field; the HTML for each version can be compared as text.

3. **Single-source of truth.** The document format is fully controlled by the
   backend template. Counsellors cannot modify the HTML; they see exactly what the
   system produced. This is appropriate for a document that may be shared with
   students or parents.

4. **Browser-native rendering.** Every modern browser renders HTML natively. No
   plugin, viewer, or library is needed on the client. Mobile browsers handle
   `<iframe>` and print dialogs correctly (REQ-052 compliance).

5. **Print equivalence.** `@media print` CSS in HTML produces print output that is
   functionally equivalent to PDF for this document type. Page breaks, margin
   control, and header/footer suppression are all achievable.

6. **WCAG AA compatibility.** HTML is the native accessibility format. ARIA roles,
   alt text for any visual elements, and semantic structure (headings, tables,
   lists) are expressible in HTML and automatically interpreted by screen readers.
   PDF accessibility requires separate effort and additional tooling.

---

## Consequences

**Accepted trade-offs:**

- **HTML is not tamper-proof.** Unlike a signed PDF, the HTML string in the database
  can be modified directly. This is acceptable for an internal tool where database
  access is controlled. If audit-grade tamper protection is required in future, a
  SHA-256 hash of the html_content should be stored alongside it.

- **`<iframe srcdoc>` has a size limit in some older browsers.** The `srcdoc`
  attribute is limited to ~2 MB in some browser implementations. For the expected
  document size (< 100 KB of HTML including inline styles) this is not a concern.
  If document sizes grow beyond 500 KB, the endpoint should instead return the
  HTML as `text/html` content type and the frontend should navigate to a dedicated
  plan URL rather than using `srcdoc`.

- **Template maintenance is a backend responsibility.** When the plan structure
  evolves (new sections, changed layout), the Jinja2 template must be updated
  and plans must be regenerated. Old plans stored in the database will reflect
  the template version that generated them; this is tracked by the `version` field.
