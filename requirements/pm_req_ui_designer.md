# Issued by: PM (Requirements Authority)

## Scope

This packet contains every requirement in the [UI] domain, plus the [FRONTEND] requirements that define which pages must exist. The UI Designer is responsible for page layout and user interaction design. The UI Designer does not write code or implement components.

---

## Page Inventory (from [FRONTEND])

The following four pages are required. No additional pages are in scope for MVP.

| REQ-ID | Page |
|---|---|
| REQ-031 | Login page |
| REQ-032 | Student List page |
| REQ-033 | Student Detail page (view and edit) |
| REQ-034 | Recommendation page |

---

## Owned Requirements

### REQ-036 [UI]
**Description:** The UI must have a clear layout. Minimal styling is acceptable; visual complexity must be avoided.
**Source:** preferences.md §5.7
**Priority:** Must Have

---

### REQ-037 [UI]
**Description:** Each recommendation displayed in the UI must show: school name, score, the reasons it matches the student, and the gaps identified for that student.
**Source:** preferences.md §5.5
**Priority:** Must Have

---

### REQ-038 [UI]
**Description:** The Recommendation page must display the action plan output (academic targets, extracurricular direction, preparation steps) alongside the school recommendations.
**Source:** preferences.md §5.6, §5.7
**Priority:** Must Have

---

### REQ-039 [UI]
**Description:** The system must not implement a complex UI/UX system (e.g., animations, advanced component libraries, design systems) in MVP.
**Source:** preferences.md §7
**Priority:** Must Have (constraint / exclusion)

---

## Page-Level Requirements Referenced from [FRONTEND]

### REQ-031 [FRONTEND] — Login Page
**Description:** The frontend must provide a Login page where counselors submit email and password credentials.
**Source:** preferences.md §5.1, §5.7

### REQ-032 [FRONTEND] — Student List Page
**Description:** The frontend must provide a Student List page that displays all students managed by the logged-in counselor.
**Source:** preferences.md §5.7

### REQ-033 [FRONTEND] — Student Detail Page
**Description:** The frontend must provide a Student Detail page that displays a student's profile fields and allows editing.
**Source:** preferences.md §5.2, §5.7
**Student profile fields to display/edit:** name, grades by subject, interests (list or tags), strengths/weaknesses (text), target region (local or international).

### REQ-034 [FRONTEND] — Recommendation Page
**Description:** The frontend must provide a Recommendation page that displays the ranked school list, scores, explanations, gaps, and action plan for a selected student.
**Source:** preferences.md §5.5, §5.6, §5.7

### REQ-035 [FRONTEND] — Generate Trigger
**Description:** The frontend must include a mechanism (e.g., a "Generate" button) that triggers the matching and recommendation generation workflow for a student.
**Source:** preferences.md §9

## Key Design Constraints

| Constraint | Source REQ-ID |
|---|---|
| Minimal styling only; no design complexity | REQ-036, REQ-039 |
| No animations, no advanced component libraries | REQ-039 |
| Primary users are counselors (not students or parents in MVP) | REQ-001, preferences.md §2 |
| System is decision-support; UI must present outputs as informational, not directive | REQ-002 |
| No mobile or desktop app layout required; web-only | REQ-001 |

---

## Content Requirements for the Recommendation Page (REQ-034, REQ-037, REQ-038)

The Recommendation page must surface the following data, all of which is generated by the backend:

**Per recommended school (up to 5 schools):**
- School name
- Score
- Explanation: why it matches (which factors contributed)
- Gaps: what the student is missing

**Action plan section:**
- Academic targets
- Suggested extracurricular direction
- General preparation steps

## Non-Goals for This Agent

- Do not write React components or any code
- Do not design beyond the four required pages
- Do not introduce role-based views or dashboards for students or parents
- Do not design a timeline or scheduling UI (not required in MVP)
- Do not design weight-tuning controls for the matching engine (weights are fixed in MVP)
